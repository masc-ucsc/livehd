Subtasks — pending

  • A — registered-output interface form -> (reg name@[0]). Repro unexpected_semantics/regout_form/impl.prp — the canonical counter2 pattern from the pyrope SKILL.md and docs/pyrope/04-variables.md: 
    pub mod top(enable:bool) -> (reg count:u8@[0]) {
  if enable { wrap count += 1 }
}
    lhd compile … --emit verilog:gen.v hard-errors: state register 'count' of 'impl.top' homes at stage 0 but its declared landing cycle 0 requires home -1. The in-body form (reg count:u8=0; value=count) works; only the interface spelling fails. Root cause: the state-register output discharge applies the feedforward invariant home.min + 1 == rec.min at upass/tolg/upass_tolg.cpp:4525 (diagnostic :4526-4534). A state register pins σ(q)=σ(din) with no flop crossing (:4434-4437, documented :4085-4086), so a counter whose din is the same-cycle increment of q has home.min=0; the +1 rule then demands home.min == -1. The comment at :4093-4094 ("discharges against home+1") directly contradicts the no-crossing model at :4085-4086. Fix: in the state-reg branch (:4514-4535) test home.min == rec.min (so @[0] matches home 0) instead of home.min + 1 == rec.min; update the comments. Before landing, confirm no passing test relies on the home+1 spelling for a state-reg output @[N>0].
  • B — hierarchical (nested-tuple) enum members. Repro unexpected_semantics/hier_enum/impl.prp — verbatim from docs/pyrope/03-bundle.md:600-616: 
    enum Animal = (,bird=(,eagle, ,parrot) ,mammal=(,rat, ,human))
pub comb top(s:bool) -> (r:u6) { r = int(Animal.bird.eagle) }   // == 0ub000011
    Hard-errors read of undefined variable 'eagle'. Root cause: nested-enum detection at inou/prp/prp2lnast.cpp:4553 calls enum_definition_node_of(e.value_node), which (:4362) returns non-null only for an explicit enum(...) node — not for the bare-tuple value (,eagle,,parrot) the spec prescribes. So the hierarchical branch is skipped, control falls into the "Expression value" branch (:4593-4605) which lowers the tuple via expr_to_node, reading eagle/parrot as variables (read_sites_ :5372) → undefined-read abort (:859-864). The explicit nested-enum(...) form already works, and the runner read side (upass/runner/upass_runner.cpp:168, :2606) already supports hierarchical enums — only the bare-tuple detection is missing. Fix: make the nested test at :4553 (or enum_definition_node_of) also accept a tuple value and parse its members with the existing process_enum_assignment logic (:4657-4699), reusing the parent-bit / __enumval machinery so inner names go to fields, not expr_to_node.
  • F — cgen emits a SystemVerilog reserved word as a signal name. Repro unexpected_bugs/reg_multi_bitslice_drop/impl.prp (variable named packed). The slice-merge logic itself is now correct, but the emitted gen.v contains reg [11:0] packed; → yosys unexpected TOK_PACKED, iverilog -g2012 syntax error. This breaks the lgyosys LEC path (it re-reads the emitted check_impl.v) and any SV-2012 consumer (verilator included). Root cause: get_scaped_name (inou/cgen/cgen_verilog.cpp:174, escape gate at :185) escapes a name to \name only if it is in the reserved_keyword set, but that set (:87-100, declared cgen_verilog.hpp:48) lists only ~14 classic Verilog-2001 words and omits the IEEE-1800 additions. packed isn't found, all its chars are legal, so the bare word is emitted. cgen does escape names with special chars (that is why \impl.top is escaped) — only reserved-word coverage is missing. Fix: extend the set at :87-100 with the IEEE-1800 list (at least packed, bit, byte, int, shortint, longint, void, type, ref, return, automatic, static, unique, priority, interface, modport, package, program, time, real). No other change needed — get_scaped_name already escapes anything in the set. Red regression: inou/prp/tests/equiv/reserved_word_signal.prp (a purely combinational -> (packed:u8) to isolate the cgen bug; + golden with an escaped \packed port) — lgcheck rejects the bare packed in the generated Verilog today; the reverse prp-v2prp-reserved_word_signal fails on the same root cause. Both go green when cgen escapes the keyword.

Subtasks — landed

landed 2026-06-20 C — tuple-typed input port read inside an if/match expression. The io interface already flattened to dotted leaves (ar.x,ar.y), but the field reads sat inside the if-arm stmts, which the SSA upass copied verbatim with copy_with_rename — leaving a tuple_get(ar,'x') with no driver. Fix (upass/ssa/upass_ssa.cpp): a new copy_branch_port_aware deep-copy for if-arms rewrites the nested port-field reads to their flattened leaf (store(dst, ref 'ar.x')), the same rewrite the top-level path already did. Regression inou/prp/tests/equiv/tup_in_port.{prp,v} (if-expr) + tup_in_match.{prp,v} (match arms); iverilog 0/256, LEC-equiv.

landed 2026-06-20 D — tuple-typed output port driven by a whole-tuple write. Two bugs: (1) the body's 2-child store(p, ___p_0) whole-tuple write was never split into the flattened leaf drives — fix (upass/ssa/upass_ssa.cpp try_rewrite_port_whole_store): split into one p.first = ___p_0.first, … per output leaf (a tuple_get per leaf, or a leaf-to-leaf ref when the source is itself a tuple input port); copy_branch_port_aware also splits per-field writes nested in if-arms. (2) even per-field leaf writes with a constant value were dropped, because is_io_output (upass/constprop/upass_constprop.hpp) compared the store name's first-level (p) against the now-flattened output entry (p.first) and never matched — a computed store survived only via demand re-emit. Fix: match a store to a flattened output leaf (name == oe.name) as well as the whole-tuple/scalar case. Regression tup_out_port.{prp,v} (whole-write) + tup_out_field_arms.{prp,v} (per-field writes in if-arms); LEC-equiv. Known gap (separate, pre-existing constprop limitation): a whole-tuple write whose source nests a runtime field inside a sub-tuple (p = (first=j, s=(m=2, n=j+3))) still errors in tolg — constprop does not fold a multi-segment tuple_get to a nested runtime leaf. Per-field nested writes (p.s.n = j+3) and const / top-level-runtime nested whole-writes all work.

landed 2026-06-20 E — signed-shift idiom emits a wrong netlist. The SHL zero-extend idiom ({N{1'b0}} | a) forces the whole ^ unsigned, and that unsigned context propagated into the bare a >>> amt term, turning the arithmetic shift logical (zero-fill) — wrong for negative a (384/1024 combos). The TODO's suggested $signed(val) >>> amt does not survive (the shift's left operand stays context-determined — verified with iverilog). Fix (inou/cgen/cgen_verilog.cpp:1086): isolate the SRA in its own self-determined signed context — $signed($signed(val) >>> amt) — but only for a genuinely signed operand (is_unsign gate; an unsigned >> must stay logical). Regression shift_combo_i8.{prp,v} (i8) + shift_sra_unsigned.{prp,v} (u8 must stay logical); iverilog 0/1024, LEC-equiv. The yosys srasll round-trip (mixed signed/unsigned, arithmetic/logical) confirms the cast is selective.

Also resolved in the index's hard-error table (tracked there, not a 2i row): overload-gathering call dispatch const add=[f1,f2]; add(x) now resolves the whole call including return binding — regressions equiv/overload_dispatch and equiv/overload_return_dispatch.
