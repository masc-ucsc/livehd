2i-issues Pyrope lowering & cgen emission defects (non-LEC)

Six independent compiler bugs surfaced by the differential test corpus in unexpected_semantics/ and unexpected_bugs/ (plus the one cgen emission bug filed under lec_missmatch/shift_combo_i8 — it is a codegen defect, not a LEC-engine issue). Each is valid, documented Pyrope that today either hard-errors in lowering or emits a wrong / invalid netlist. Pick any one and work it end-to-end from its subtask below — they have no ordering between them.

Status 2026-06-20 (adversarial re-test): E (signed-shift idiom) is now resolved — five remain open (A, B, C, D, F). Each open construct now ships a checked-in red regression under inou/prp/tests/equiv/ (tuple_in_port, tuple_out_port, reserved_word_signal); they turn green as each fix lands.

Scope. This page deliberately excludes the LEC-engine precision issues (cvc5 false-refutes on hotmux / FSM-state / memory, the \x01nxt: phantom-output, the lgyosys no-reset-flop disagreement). Those are checker bugs, not compiler bugs, and live in the repo-root issues.md (Group C). Every code pointer below was re-verified by reading the source on 2026-06-20 against a fresh bazel build -c dbg //lhd:lhd.

Subtasks — pending

  • A — registered-output interface form -> (reg name@[0]). Repro unexpected_semantics/regout_form/impl.prp — the canonical counter2 pattern from the pyrope SKILL.md and docs/pyrope/04-variables.md: 
    pub mod top(enable:bool) -> (reg count:u8@[0]) {
  if enable { wrap count += 1 }
}
    lhd compile … --emit verilog:gen.v hard-errors: state register 'count' of 'impl.top' homes at stage 0 but its declared landing cycle 0 requires home -1. The in-body form (reg count:u8=0; value=count) works; only the interface spelling fails. Root cause: the state-register output discharge applies the feedforward invariant home.min + 1 == rec.min at upass/tolg/upass_tolg.cpp:4525 (diagnostic :4526-4534). A state register pins σ(q)=σ(din) with no flop crossing (:4434-4437, documented :4085-4086), so a counter whose din is the same-cycle increment of q has home.min=0; the +1 rule then demands home.min == -1. The comment at :4093-4094 ("discharges against home+1") directly contradicts the no-crossing model at :4085-4086. Fix: in the state-reg branch (:4514-4535) test home.min == rec.min (so @[0] matches home 0) instead of home.min + 1 == rec.min; update the comments. Before landing, confirm no passing test relies on the home+1 spelling for a state-reg output @[N>0].
  • B — hierarchical (nested-tuple) enum members. Repro unexpected_semantics/hier_enum/impl.prp — verbatim from docs/pyrope/03-bundle.md:600-616: 
    enum Animal = (,bird=(,eagle, ,parrot) ,mammal=(,rat, ,human))
pub comb top(s:bool) -> (r:u6) { r = int(Animal.bird.eagle) }   // == 0ub000011
    Hard-errors read of undefined variable 'eagle'. Root cause: nested-enum detection at inou/prp/prp2lnast.cpp:4553 calls enum_definition_node_of(e.value_node), which (:4362) returns non-null only for an explicit enum(...) node — not for the bare-tuple value (,eagle,,parrot) the spec prescribes. So the hierarchical branch is skipped, control falls into the "Expression value" branch (:4593-4605) which lowers the tuple via expr_to_node, reading eagle/parrot as variables (read_sites_ :5372) → undefined-read abort (:859-864). The explicit nested-enum(...) form already works, and the runner read side (upass/runner/upass_runner.cpp:168, :2606) already supports hierarchical enums — only the bare-tuple detection is missing. Fix: make the nested test at :4553 (or enum_definition_node_of) also accept a tuple value and parse its members with the existing process_enum_assignment logic (:4657-4699), reusing the parent-bit / __enumval machinery so inner names go to fields, not expr_to_node.
  • C — tuple-typed input port. Repro unexpected_semantics/tup_in_port/impl.prp — verbatim from docs/pyrope/06-functions.md:357: 
    pub comb top(ar:(x:u3, y:i4), cond:bool) -> (res:i5) {
  res = if cond { ar.x + 1 } else { ar.y - 1 }
}
    Hard-errors upass.tolg: unresolved reference '___res_1' — it has no driver. Root cause (confirmed from the post-upass LNAST): the port is correctly flattened — io_meta().inputs = ar.x, ar.y, cond, so the leaf input pins exist (upass/tolg/upass_tolg.cpp:183-184). But the body reads them as tuple_get(___res_1, ar, 'x') — a field read of the whole tuple ar, which has no tuple driver (only the leaves do). lower_tuple_get (:2207) can't resolve ar.x, so ___res_1 is never bound and the undriven check fires at :414. Fix: bind a tuple_get whose base is a tuple-typed input directly to the flattened leaf input pin (ar.x), or have upass/detuple rewrite the field reads to dotted-leaf refs (the same flattening it already does for struct vars/regs). Pairs with D. Red regression: inou/prp/tests/equiv/tuple_in_port.prp (+ golden .v) — fails at the prp→verilog step today, goes green when this lowers.
  • D — tuple-typed output port driven by a whole-tuple write. Repro unexpected_semantics/tup_out_port/impl.prp (documented in docs/pyrope/06-functions.md): 
    pub comb top(c:bool) -> (p:(first:u8, second:u8)) {
  p = (first=1, second=2)
}
    Hard-errors output 'p.first' is never driven by the body of 'impl.top'. Root cause (confirmed from LNAST): io_meta().outputs = p.first, p.second, but the body builds ___p_0 = tuple_add(first=1, second=2) then store(p, ___p_0) — a whole-tuple write to p. tolg records a driver for p/___p_0 but not for the flattened leaf names, so the output finalize loop (upass/tolg/upass_tolg.cpp:260-283) finds no pin_map_ entry for p.first → error at :277. Fix: split a whole-tuple write to a tuple-typed output into per-leaf drives (p.first←___p_0.first, etc.) matching the flattened output names — in upass/detuple (it already classifies whole-write tuples at upass_detuple.cpp:517) or in tolg's output binding. Pairs with C. Red regression: inou/prp/tests/equiv/tuple_out_port.prp (+ golden .v) — fails at the prp→verilog step today (per-field writes fail the same way), goes green when the whole-tuple write drives the flattened leaves.
  • resolved 2026-06-20 E — cgen emits a wrong netlist for a signed-shift idiom. Fixed exactly as proposed below (the SRA term now emits $signed($signed(a) >>> amt)): 0/1024 iverilog mismatches and PROVEN equivalent under both cvc5 and lgyosys. Passing regression inou/prp/tests/equiv/shift_combo_i8.prp (+ golden). Repro lec_missmatch/shift_combo_i8/impl.prp (a cgen bug, filed there only because it also makes the two LEC engines disagree): 
    pub comb top(a:i8, b:u2) -> (r:i8) { wrap r = (a << b) ^ (a >> b) }
    Independent iverilog sim shows the emitted gen.v differs from spec in 384/1024 input combos (e.g. a=0x80,b=1: gen=0x40, correct=0xC0). cvc5 (reasons on the LGraph) says EQUIV — i.e. the IR is correct; lgyosys (reasons on gen.v) correctly refutes. So it is purely a Verilog-text bug. cgen emits xor_28 = (({12{1'b0}} | a) << amt) ^ (a >>> amt); Root cause: the SHL operand is forced unsigned by the deliberate zero-extend idiom ({N{1'b0}} | a) (inou/cgen/cgen_verilog.cpp:1068), while the SRA term a >>> amt (:1083-1086) is bare and relies on a being signed; they are joined by the generic add_expression (:237-245, used at :1099-1108) with no per-operand $signed/$unsigned. Verilog evaluates ^ in unsigned context if any operand is unsigned, and that context propagates into the >>> term, turning the arithmetic shift into a logical (zero-fill) one → wrong for negative a. Fix: make the SRA term self-contained — emit $signed(val) >>> amt at :1086. (Do not remove the SHL zext; the broad fix is an explicit signedness cast on every operand of the generic binary-op path.)
  • F — cgen emits a SystemVerilog reserved word as a signal name. Repro unexpected_bugs/reg_multi_bitslice_drop/impl.prp (variable named packed). The slice-merge logic itself is now correct, but the emitted gen.v contains reg [11:0] packed; → yosys unexpected TOK_PACKED, iverilog -g2012 syntax error. This breaks the lgyosys LEC path (it re-reads the emitted check_impl.v) and any SV-2012 consumer (verilator included). Root cause: get_scaped_name (inou/cgen/cgen_verilog.cpp:174, escape gate at :185) escapes a name to \name only if it is in the reserved_keyword set, but that set (:87-100, declared cgen_verilog.hpp:48) lists only ~14 classic Verilog-2001 words and omits the IEEE-1800 additions. packed isn't found, all its chars are legal, so the bare word is emitted. cgen does escape names with special chars (that is why \impl.top is escaped) — only reserved-word coverage is missing. Fix: extend the set at :87-100 with the IEEE-1800 list (at least packed, bit, byte, int, shortint, longint, void, type, ref, return, automatic, static, unique, priority, interface, modport, package, program, time, real). No other change needed — get_scaped_name already escapes anything in the set. Red regression: inou/prp/tests/equiv/reserved_word_signal.prp (a purely combinational -> (packed:u8) to isolate the cgen bug; + golden with an escaped \packed port) — lgcheck rejects the bare packed in the generated Verilog today; the reverse prp-v2prp-reserved_word_signal fails on the same root cause. Both go green when cgen escapes the keyword.
