Memory-invariant templates (old V8 Attack A; moved from 2f-verify 2026-07-12)

  • Why memories explode: each cycle appends stores; read-over-write chains grow with the bound; the solver case-splits on address aliasing across the whole history.
  • Attack: per-entry invariants via comptime-unrolled for loops in formal blocks + template instantiation over the memories in the hard asserts' cones (valid ⇒ field-in-range per entry, entry-count == popcount of valids, …), targeted where timeout cores show store chains are the kernel. Most memory-heavy asserts need a per-entry property, not the contents. (The CEGAR lazy-read alternative was dropped; an assumed invariant constrains the reads directly, which is the same synergy without the machinery.)
  • Near-term mitigation (why this can wait): the landed formal.minetimeout timeout-core diagnosis already points the agent at the store-chain kernel, and a coding agent can hand-write the per-entry invariants in a sidecar block today; this item is the AUTOMATED template instantiation (register/state templates stay in 2f-verify's mining Source 2).
