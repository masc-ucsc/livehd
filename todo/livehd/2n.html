Command-line testing

The server is plain JSON-RPC over stdio (Content-Length framing), so every feature is exercisable from a shell — pipe framed requests into lhd pyrope lsp and read the framed replies, no editor required. Two harnesses live in lhd/tests/:

  • Automated regression — lhd/tests/lhd_lsp_test.py (bazel //lhd/tests:lhd_lsp_test). Drives a full lifecycle (initialize negotiates utf-8 + pull diagnostics, didOpen/didChange/didSave/didClose), asserts a broken .prp yields an error diagnostic with a span and a fixed buffer clears it, and that an unknown request returns -32601. Each new phase extends this file: (1) assert initialize now advertises the phase's provider flag, (2) fire the request against a known buffer, (3) assert the response shape. It is the regression gate for B–D.
  • Interactive driver — lhd/tests/lsp_cli.py. Fires ONE feature request at a .prp buffer and prints the raw JSON reply (after first printing the negotiated capabilities, so you can see which providers are on). It runs the whole initialize → didOpen → request → shutdown → exit session for you and exits non-zero when the server answers MethodNotFound — i.e. the feature isn't wired yet. LINE/COL are 0-based LSP wire coordinates (line 0 = first line); the binary is $LHD, else ./bazel-bin/lhd/lhd.

Design notes

Salvaged from docs/contracts/pyrope_lsp.md (now retired)

A Language Server for Pyrope only (.prp) built on LiveHD's existing front-end. lhd pyrope lsp runs the server over stdio (lifecycle + textDocumentSync Full + didOpen/didChange/didSave/didClose + publishDiagnostics push and textDocument/diagnostic pull, with positionEncoding negotiation). Diagnostics come from inou.prp |> pass.upass — analyze() runs prp2lnast (buffer-backed ctor), then Pass_upass::work with tolg:0 toln:0 (semacheck / typecheck / constprop / bitwidth / verifier / assert …), collecting every stage's records from the shared core/diag in-memory sink. Library: lsp/ (livehd::lsp::run_stdio). Design contract for this task. Builds on the landed diagnostics foundation (core/diag), shares the JSONL/step model with the agent CLI (the LSP folds under livehd lsp; see 3y-agent), and inherits source spans from 1f (landed 2026-06-10) / the LiveHD docs (precise for syntax today, sharpening as the 1f (landed 2026-06-10)/3f (landed 2026-06-11) spans propagate).

Claude Code integration facts below were verified against the in-session LSP tool definition and the official docs (code.claude.com/docs/en/tools-reference, /plugins-reference, /vs-code, /hooks); LSP wire facts against the LSP 3.17 spec. 1. Goal

A Language Server for Pyrope only (.prp) built on LiveHD's existing front-end. It must never touch the Verilog/Yosys path — .v/.sv are out of scope.

Design priorities, in order:

  1. Primary consumer: Claude Code. Target the exact LSP surface Claude Code's built-in LSP tool consumes (§2). Implement that subset first.
  2. Secondary consumer: neovim (vim.lsp / nvim-lspconfig).
  3. One copyable binary. A single livehd-lsp (later also a livehd lsp subcommand of the agent CLI) registerable with an editor.
  4. Reuse, don't rebuild. Diagnostics from the landed core/diag Sink; parsing from the shared tree-sitter-pyrope grammar inou/prp already links; types/widths from upass/typecheck + upass/bitwidth; name maps seeded from the ones prp2lnast already builds.

Scope. This contract covers the LSP server only. The tree-sitter-pyrope grammar + neovim queries, and the prpfmt formatter (including textDocument/formatting), are a separate project — treated here as external, sharing only the grammar. See §8 for the boundary.

5. The fast pipeline (latency is the constraint)

Re-analysis runs on (almost) every save, so the pipeline is cheap and Pyrope-only, stopping well before LGraph lowering. To surface every diagnostic the CLI would (the user's requirement), analyze() runs the inou.prp |> pass.upass chain in-process:

prpparse (streaming parse_next) → prp2lnast (buffer-backed ctor) → converter.get_lnast()
  → Eprp_var var; var.add(lnast); var.add("tolg","false"); var.add("toln","false")
  → Pass_upass::work(var)        (default order MINUS the coalescer — under toln:0 it is
                                  dropped, nothing consumes the rewritten LNAST: verifier,
                                  semacheck, attributes, typecheck, constprop, bitwidth,
                                  assert, verifier; tolg:0 = no LGraph/cgen/Yosys, toln:0 =
                                  don't materialize the rewritten LNAST. When LSP is enabled
                                  a flag-gated `lsp` pass is appended that builds the
                                  semantic-index map during this same traversal.)
  → STOP

No lnastfmt stage (goal 1h, 2026-06-05): its user-facing checks moved into upass/semacheck (read-only-attr writes, declare-once/shadowing — located spans via the declare/store/attr_set loc-carry chain), so lnastfmt is a compiler-internal structural validator with nothing to tell an end user. Each stage is wrapped in try/catch: a Pass::error throw has already flushed its record into the sink (parser_error_int → sink.flush()), and the catch keeps the LSP alive. (Earlier drafts ran only a reduced {attributes, typecheck, bitwidth} order; the full pass.upass is the landed behavior so the LSP and the CLI report identically.)

Two tiers:

  • Syntax tier (pending; re-specced 2026-07-08 against the prpparse front-end) — a bare prpparse parse loop: construct prpparse::Source_buffer + prpparse::Parser over the buffer and drain parse_next() to end of input — lex/parse only, no LNAST lowering, no upass pipeline — catching prpparse::Parse_error and mapping its Diag (shape-compatible with core/diag) straight to the LSP diagnostic. prpparse is streaming and fail-fast with no ERROR-node phase (inou/prp/prp2lnast.cpp:221 — check_parse_errors() is moot; a returning parse is well-formed), so the tier yields at most the first syntax error, near-instantly and with bounded memory (the CST arena recycles per construct). Run it on didChange; the full compile (semantic tier) runs on idle/didSave. Two costs to carry over: Prp2lnast's pre-parse nesting-depth guard must be duplicated (or accepted as absent), and the semantic index is only rebuilt by the semantic tier — hover/nav stay empty (never stale: the index is version-tagged and dropped on didChange) until the next semantic run. The old design — reintroduce the tree-sitter-era parse_only ctor flag that stopped after check_parse_errors() (removed 2026-06-04) — is dead: no ctor seam is needed, the facade's streaming API is directly callable.
  • Semantic tier — the full front-end above. Landed behavior today: run directly for every didOpen/didChange/didSave, no debounce. When the syntax tier ships, didChange demotes to syntax-only and this tier moves to idle/didSave. Workspace indexing is a separate background process, so it does not share the foreground front-end's process-global pass/diagnostic state.

Partial semantic results (by construction): the primary consumer is a coding agent editing a mid-error buffer, so this is the common case, not the exception. Because the lsp pass populates the index map incrementally per statement during the single upass traversal, a fatal error simply stops the traversal: the map holds correct kind/type/range for every statement before the first error and is empty after it (the error is reported as the diagnostic). Hover/navigation answer from that partial map for the earlier statements. There is therefore no "collect-and-continue" front-end change — "empty after the first error" is the accepted contract. The index is tagged with the textDocument version and dropped on every didChange, so the LSP never answers from an unrelated stale version.

Compiler/LGraph state remains ephemeral: no @tag or lgdb. The derived, immutable LSP semantic index is deliberately persistent under the XDG cache as described in §12.

Text sync is a fixed Phase-A decision, not an open question. A diagnostics-only server still MUST advertise textDocumentSync or it defaults to None and receives no didChange/didSave content — the whole pipeline never fires. Advertise:

textDocumentSync = { openClose: true, change: TextDocumentSyncKind.Full (=1),
                     save: { includeText: true } }

Full sync (whole buffer each change) matches the string-rebuild driver; Incremental (ranged deltas + a server-side buffer copy) is explicitly out of scope.

Span fidelity degrades gracefully but the current state is not uniform: upass/bitwidth emits fully null spans; upass/typecheck is null except cassert/func_call diagnostics, which already carry file + start_line (line only, no start_byte) via the loc-carry chain; inou/prp syntax diagnostics carry precise byte spans. 7. Position encoding & indexing — the classic footguns

  • Byte → Position. core/diag Span stores bytes; LSP Position is 0-based line + 0-based character. Put the one byte→(line, character) resolver in core/diag (it is also what to_text needs for the carat) so the text renderer, this LSP, and any future transport share it.
  • Encoding is negotiated, not chosen. LSP 3.17 defaults character to UTF-16 code units. The server MUST read InitializeParams.capabilities.general.positionEncodings; if it offers "utf-8", return positionEncoding: "utf-8" in InitializeResult.capabilities — otherwise return "utf-16" (the mandatory default and the only legal value when the client lists none). neovim supports utf-8/16/32. The negotiated encoding governs every Position/Range (diagnostics, hover, documentSymbol, definition/references, callHierarchy), so the byte→Position resolver must be parameterized by the negotiated encoding, not static.
  • Wire vs Claude Code's tool surface. Claude Code's LSP tool takes 1-based line/character "as shown in editors" — that is the tool's contract to the agent, not the wire. The server speaks standard 0-based LSP; the +1/-1 conversion lives entirely in Claude Code's client and the server must never apply it. (Verified accurate.)

8. Scope boundary — LSP server vs the separate tree-sitter/prpfmt project

Three artifacts share one grammar but are owned by two projects:

   tree-sitter-pyrope  ── grammar + queries/*.scm   ┐  SEPARATE PROJECT
   prpfmt              ── C formatter over the tree  ┘  (neovim grammar/queries + formatting)
   livehd-lsp          ── SEMANTIC layer: diagnostics + hover + nav   ← THIS contract (2n)

This contract (2n) is the livehd-lsp semantic server only. The grammar + neovim queries and the prpfmt formatter are a separate project; 2n treats them as external and depends on exactly one thing from them: the shared tree-sitter-pyrope grammar, which inou/prp/prp2lnast already link and the LSP reuses (one parse, one TSTree per buffer). 2n does not ship neovim queries, does not embed prpfmt, and does not implement textDocument/formatting.

Relationship to the separate project:

  • Formatting is delivered there, not here. neovim formats via conform.nvim → prpfmt (or the formatter's own LSP if that project grows one); Claude Code formats via a PostToolUse hook that runs the separate project's prpfmt binary on .prp edits (Claude Code's LSP tool never calls formatting).
  • Grammar = the only shared seam. If the separate project bumps the grammar, the LSP picks it up through prp2lnast's parser; nothing else couples.

Handoff notes for the separate project (already-verified findings, recorded here so they aren't lost — not 2n work):

  • queries/{highlights,indents,folds}.scm were regenerated 2026-06-02 against the current grammar (HEAD 795f93f) and validated with node_modules/.bin/tree-sitter query. The old set referenced ~22 defunct node names; the grammar now emits function_call_expression/tuple/if_expression/function_definition_decl and wraps every operator in a named op_*/reduction_* node. nvim-treesitter fails the whole file on one unknown node type, so add a CI check asserting every (node) in the queries exists in src/node-types.json.
  • prpfmt is embeddable (no file-scope globals; output via st->outfile FILE*; entry print_description→prpfmt_solve→prpfmt_render→prpfmt_free_buffer over a PrpfmtState; capture via open_memstream; refuses ERROR/MISSING trees). Link prpfmt.c+ir.c ONLY (never its src/parser.c — duplicate tree_sitter_pyrope() link clash); the PyropeSymbol enum in prpfmt.h is a hand-copied parser.c snapshot that must be regenerated on grammar bumps.

9. Packaging / install & the two Claude-Code diagnostics channels

Landed 2026-07 — partially superseded. The shipped recipe (README.md, "Claude Code" / "Neovim (0.11+)" sections) uses a skills-directory plugin with inline lspServers in plugin.json plus the checkout-aware scripts/prplsp wrapper on $PATH — not the plugin-bundled binary sketched below. The notes below stay for the bundled-binary variant and the two-channel analysis, which remain accurate.

  • Binary: one livehd-lsp (the semantic server; no prpfmt — that ships with the separate project).
  • Claude Code — bundle, don't require $PATH. The .lsp.json command supports ${CLAUDE_PLUGIN_ROOT} / ${CLAUDE_PROJECT_DIR} substitution and an env map, so ship the binary inside the plugin: 
    { "pyrope": {
    "command": "${CLAUDE_PLUGIN_ROOT}/bin/livehd-lsp",
    "args": [], "transport": "stdio",
    "extensionToLanguage": { ".prp": "pyrope" } } }
    The key is the server name; command is the binary; extensionToLanguage maps the extension to an LSP languageId (not to the binary). ${CLAUDE_PLUGIN_ROOT} changes on plugin update (old dir kept ~7 days) — don't store state there. Reserve a bare-name-on-$PATH command (via a wrapper, not a bazel-bin/... path) for system installs. Optional PostToolUse hook for prpfmt-on-save: matcher "Edit|Write" matches by tool name only, so the hook script must guard on the .prp extension itself and read the path from .tool_input.file_path on stdin.
  • Two diagnostics channels in Claude Code (the design must not treat "Claude Code" as monolithic):
    1. Bare CLI / neovim — the LSP tool drives our .lsp.json-registered livehd-lsp directly. This is the design's target.
    2. VS Code / Cursor / JetBrains — Claude reads the editor's Problems panel via the built-in ide MCP server tool mcp__ide__getDiagnostics (connected by /ide), independent of .lsp.json. In an IDE already running livehd-lsp as its editor LSP, diagnostics reach Claude through mcp__ide__getDiagnostics instead of our plugin. The plugin still serves case 1 and any editor without a native Pyrope LSP.
  • neovim: point the built-in vim.lsp.config/vim.lsp.enable (neovim 0.11+, superseding parts of nvim-lspconfig) at livehd-lsp. Tree-sitter registration + queries and conform.nvim → prpfmt are configured by the separate grammar/formatter project, not here.

12. Workspace/index decisions

Trimmed 2026-06-19: Phase D ships an in-memory session index (lazily built on the first workspaceSymbol, kept for the session; open buffers always override). The persistent on-disk machinery below — two-process indexer, XDG cache, content-addressed manifest, .gitignore/symlink walk — is deferred to Phase E behind a measured-need gate. Bullets tagged (Phase E) do not block D.

  • Workspace roots: consume InitializeParams.workspaceFolders, falling back to legacy rootUri, and handle workspace/didChangeWorkspaceFolders.
  • Discovery: recursively discover .prp files under each workspace, honor .gitignore, do not follow directory symlinks, and exclude generated/build trees such as .git, bazel-*, and node_modules. An open buffer always overrides the disk version.
  • Session index (Phase D): on the first workspaceSymbol, walk the discovered .prp files, analyze each through the same in-process pipeline, and keep the derived per-file indexes in RAM for the session. An open buffer always overrides its disk version. Re-analyze a file in the foreground on first query if it was edited on disk. Because diag::sink() is a process-global static, these foreground analyses are sequential, not concurrent threads.
  • (Phase E) Two processes, not two analysis threads: a separate one-shot background indexer process walks workspace files sequentially, writes their derived indexes, then exits. Terminate an in-flight indexer when workspace folders change or the LSP exits. This isolates process-global front-end state such as diag::sink().
  • (Phase E) Persistent cache: store it below $XDG_CACHE_HOME/livehd/lsp/ (with the platform cache fallback when XDG_CACHE_HOME is unset), keyed by canonical workspace path. Validate every entry with its source-content hash plus LiveHD/index-schema version. Store derived semantic metadata, not source text.
  • (Phase E) Concurrent writes: use immutable/content-addressed per-file entries and atomic replacement; the background indexer may atomically publish a fresh workspace manifest without corrupting foreground results. Persist a foreground result immediately after analysis rather than relying only on clean LSP shutdown.
  • (Phase E) Foreground reads from cache: load the workspace manifest at startup. For a later unopened file request, use a matching cached entry; if absent or stale, analyze that file in the foreground, answer, and cache the result.
  • Change scheduling: Full text sync and push+pull diagnostics remain fixed. Run semantic analysis on every open/change/save without debounce.

Formatting / neovim queries remain out of scope (§8).
