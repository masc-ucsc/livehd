1b Bundle storage — split attributes into a second flat map, drop the __ prefix, bound the prefix scans

Bundle (lnast/bundle.{hpp,cpp}) is the comptime value/tuple representation the upass runner threads through every pass. Today it stores everything — attributes, named fields, and positional fields — in a single std::map<std::string, Entry, Canonical_less> keyed by a flat dotted path (foo.bar.__bits, 0.__enumentry, name, …). Attributes are told apart from data only by a __ prefix on the last segment, so every operation re-sniffs that prefix (is_attribute is called inside nearly every walk). Separately, most prefix-scoped operations — get_bundle, has_bundle, del_int, is_ordered, the concat helpers — scan the whole map even though the keys they care about live in one contiguous run.

This task makes two narrow, low-risk changes and keeps the flat dotted-key data map exactly as it is:

  • Split attributes into a second flat map and drop the __ prefix. Data leaves (a.b, a.c, a.0, 0, …) stay in the primary map; attribute leaves move to a sibling map keyed by bare name. Membership in the attr map is what makes a key an attribute — the per-key __ string-sniff disappears, and a data field literally named __x can no longer collide with the attribute namespace.
  • Replace the full-map scans with lower_bound-bounded prefix walks. Because the data map is sorted by Canonical_less, every key under a top segment forms one contiguous run; lower_bound(seg) + walk-until-out-of-prefix turns the O(N) scans into O(log N + k), where k is "one field's leaves" — a couple of entries.

What this task deliberately does NOT do: it does not introduce a recursive node, and it does not change the flatten-to-dotted-leaf-keys contract. Two bundles are still equal iff their flattened {leaf-path → value} maps are equal (the dotted key carries the full path, including the decimal index for positionals — ((1,2),3) ⇒ 0.0,0.1,1 vs (1,2,3) ⇒ 0,1,2, correctly unequal). So compare_bundles_eq, structural_does, the does machinery, and the LGraph/Verilog port-naming all ride on the unchanged data map.

Depends on

None — internal IR refactor. The concat / scalar-collapse / promotion semantics it preserves are LEC-pinned by the existing tuple tests, so it can land independently of 1d / 1r (both also touch upass tuple handling — sequence to avoid churn, but no hard ordering).

Target shape

Each Bundle keeps the flat dotted-key storage but holds two maps, both ordered by the same Canonical_less (bare names throughout — no __):

  • std::map<std::string, Entry, Canonical_less> data — named and unnamed leaves by dotted path. Named first-segments stay as-is (name), unnamed are bare decimals (0, 1, …). The scalar case is just a single-entry data map (the today's "0"/scalar leaf) — no separate scalar field is introduced.
  • std::map<std::string, Entry, Canonical_less> attrs — attribute leaves, keyed field_path.attr_name with the attribute name as the last segment (a.b.bits, not a.b.__bits). A whole-bundle attribute keys on the bare attr name alone (bits, replacing __bits / 0.__bits).

Canonical order (named-alpha < unnamed-numeric) stays implicit in data's iteration; the attrs/data split removes the need to interleave the two categories or sniff them apart. Canonical_less is kept and load-bearing — it both keeps unnamed numbering numeric (2 before 10, not lexicographic) and makes the prefix runs contiguous so lower_bound works. Using the same comparator on both maps keeps a field's data run and its attr run aligned. Do not fall back to plain std::string ordering.

The reshape is narrowly about killing the __ overlap and the redundant scans — not a redesign. The legacy dotted-key methods are fine and stay. Today the sorted-iteration views (entries()/non_attr_entries()/get_attrs()/top_levels()) materialize a fresh std::vector on every call (bundle.hpp: "Materialized on each call") and the prefix-scoped lookups walk the whole map. With attributes already separated and the data map sorted, those views become direct bounded walks of one or both maps, and hot consumers iterate the bounded range instead of paying a per-call allocation or a full scan.

Design notes

The __ alias bug this fixes. Because attrs and data share one keyspace, a data field literally named __x would land in the attribute namespace, and the detector has to special-case ___ (triple underscore) to avoid false positives (bundle.hpp:232, bundle.cpp:373,618). With a separate attrs map the namespaces no longer overlap — a field bits and an attribute bits coexist cleanly. The Pyrope-facing doc rule already says attributes have no __ prefix (pyrope_bundle_attribute_rules); this brings storage in line with the contract.

Attribute names actually in flight (full set, so A can be exhaustive): bits, sbits, ubits, min, max, dp_assign, enumentry, pub, pub_unit. Note ubits/sbits appear only in bitwidth.cpp's suffix checks and may be stale vs. the doc rule (size = bits/max/min only) — confirm and prune while there.

Why keep flat (and not recurse). The earlier draft of this task proposed a recursive node with three containers (attrs / named-map / unnamed-vector). Its two payoffs both evaporate once you bound the scans: the O(N)→O(log N + k) sub-bundle access comes from lower_bound on the sorted flat map, and the zero-copy iteration comes from walking the bounded map range directly instead of materializing a vector. A recursive node would also re-introduce, at every level, the category/numeric comparator the vector was meant to avoid — and would fork the flatten-compare contract that compare_bundles_eq / does / Verilog naming all depend on. Flat storage is lossless w.r.t. tree structure (the dotted key encodes the full path), so there is no semantic reason to recurse.

lower_bound boundary caveat. The stop condition is a segment boundary, not a string prefix: a must bound a and a.b but not ab/abc. Reuse match_first_partial as the terminator (it already encodes the .-boundary rule); lower_bound only gets you to the start of the run cheaply. The bare scalar key 0 sorts immediately before 0.foo (a strict prefix sorts first), so scalar carriers and their sub-leaves stay in one contiguous run.

Behavior that must round-trip exactly (the risky corners — all currently LEC-verified; most get simpler once attrs are split out):

  • Scalar carriers with attrs. add_int copies attribute leaves under a scalar key so get_bundle(key) round-trips the parse-time __enumentry tag even though is_scalar() ignores attrs. In the new shape this is "data has key a, attrs has a.enumentry" — the scalar fast path can no longer mistake the attr leaf for a data leaf.
  • Scalar→.0 promotion. The set() promotion pass that rewrites a scalar slot to slot.0 when a deeper insert arrives (and the symmetric concat promote_scalar_to_sub) now operates on data only — the mid-walk is_attribute special-casing (bundle.cpp:654, 662–665) drops out because attrs are no longer interleaved.
  • concat merge rules. attrs = rhs-wins; named = per-name merge with nil-drop and scalar-pack into a growing sub-tuple; unnamed = append at the next free index. Preserve precisely — these are pinned by the ++ / bundle-literal tests. The attr merge now writes the attrs map; the named/unnamed merge writes data.
  • Single-element collapse. (c=3) == 3 — a one-entry bundle reads as its scalar while still carrying the one field. This is a Pyrope language rule and stays in upass_constprop.cpp (the 1-entry-tuple cases), not in Bundle — it is not pure flat-key-set equality and must not migrate into the storage layer.

Doc reconciliation (unnamed-index semantics). The flat data map keeps the rule "unnamed index counts only among unnamed entries; named fields are name-access-only and consume no positional slot" — consistent with the authoritative (a=(b=1,c=2),7,3,d=3) == (a.b=1,a.c=2,d=3,7,3) example and with current behavior. docs/pyrope/03-bundle.md still carries a stale counter-example (mut a=(b:u8, b, b:u8=nil, const c=4) // a[2]==a.b, asserted equal to an all-positional (100,100,200,4)) that implies named fields consume positions — which compare_bundles_eq would not fold true. Reconcile/fix that example as part of this task so the doc matches the storage.

Blast radius (small, scoped). Attribute-API callers: inou/prp/prp2lnast.cpp (10× has_attr), upass/constprop/upass_constprop.cpp (get_attr/set_attr ×4, get_attrs ×2), upass/core/upass_core.hpp, lnast/tests/symbol_table_test.cpp. Iteration consumers: constprop + the symbol-table test only. __ hard-codes outside the bundle: bitwidth.cpp, cgen_verilog.cpp. The public API stays stable (the attr-map split is internal), so the cross-file diff is just the two external suffix-matchers + the test; the lower_bound work (B) and view collapse (C) are internal to bundle.cpp plus the constprop reader.
