1b Bundle/Entry storage — move to upass/core/, promote the always-present pass facts to typed Entry fields, split residual attributes, bound the prefix scans

Bundle (lnast/bundle.{hpp,cpp}) is the comptime value/tuple representation the upass runner threads through every pass. Today it stores everything — attributes, named fields, and positional fields — in a single std::map<std::string, Entry, Canonical_less> keyed by a flat dotted path (foo.bar.__bits, 0.__enumentry, name, …). Attributes are told apart from data only by a __ prefix on the last segment (is_attribute is called from ~26 sites inside the walks). Most prefix-scoped operations — get_bundle (bundle.cpp:495), has_bundle (:476), del_int (:343), is_ordered (:932), get_scalar_name, the erase_top/next_sub_pos lambdas inside concat — scan the whole map; the iteration views (entries()/non_attr_entries()/get_attrs()/top_levels()) materialize a fresh std::vector on every call. Entry is {bool immutable; Const trivial} (bundle.hpp:14–25), and key_map is declared mutable (bundle.hpp:95). All verified 2026-06-09.

Reshaped 2026-06-09 (design review with 2b). The earlier draft of this task was "two narrow changes, keep everything else". The 2b push-based migration made a stronger shape obvious, so 1b now makes four changes:

  • Move Bundle + Symbol_table to upass/core/. They are upass-domain structures (the per-pass fact record and its scope table), and the typed fields below must stay in sync with the upasses — lnast/ should never have known about bw_max. Verified clean: the only outside users are lnast/lnast_builder.cpp (static dotted-key string helpers only — those stay behind in lnast/ as a small key-utils header) and a vestigial #include "symbol_table.hpp" in inou/prp/prp2lnast.hpp:18 (its only st is a local Lnast_nid) — delete it.
  • Promote the always-present pass facts to typed fields. The facts every pass generates and reads (kind, declared max/min, bw_max/bw_min, comptime; mode and typename at the Bundle level) stop being string-keyed attr entries and become typed fields on Entry (enum/Const members). Per-field facts get the same first-class treatment as root-scalar facts — t.a:u4's kind/envelope/bw live in t's Entry for leaf a, no path.attr string keys, and 2b's store path ("bitwidth copies the selected field's bw_max/bw_min") becomes an Entry-to-Entry member copy. The 2b per-pass attribute table is the authoritative field registry: one writer per field; adding a fact = one struct field + one table row.
  • Split the residual attributes into a second flat map and drop the __ prefix. After the promotion, the attrs map holds only the genuinely rare/dynamic attributes — sticky _foo (with debug normalized to _debug at set time), enumentry, pub/pub_unit, dp_assign, the category-B pin-wiring attrs (clock_pin, posclk, …). Very few entries per bundle → a plain sorted std::map is fine. Membership in the attrs map is what makes a key an attribute — the per-key __ string-sniff disappears. Stickiness is encoded in the canonical attr name (leading _): no per-entry bit, no third map, and since _ sorts apart from the letter-named attrs, sticky_attributes() is a bounded walk of one contiguous run.
  • Inline root Entry — scalars allocate zero map nodes. Most variables have no fields: the root scalar's Entry becomes an inline member, and the data map holds tuple leaves only (the "0" scalar-leaf key retires). A scalar variable, a compiler tmp, or a 2b make_const wrapper is a single allocation with both maps empty.

Depends on

None — internal IR refactor. The concat / scalar-collapse / promotion semantics it preserves are LEC-pinned by the existing tuple tests. 2b is a flag-day rewrite that consumes subtasks D/E below (and the upass/core/ location from A), so 1b must be green before 2b starts.

Target shape

// upass/core/kind.hpp — shared lattice, moved out of uPass_typecheck
enum class Kind : uint8_t { unknown, integer, boolean, string, range,
                            tuple, nil, enumv, type_name };

// upass/core/bundle.hpp
struct Entry {                 // one per leaf: the inline root scalar, or a tuple leaf
  Const value;                 // today's `trivial`; Dlop::invalid() = unknown
  Const decl_max, decl_min;    // declared envelope (type-bake pre-step; persistent)
  Const bw_max,   bw_min;      // derived range (bitwidth; per-write)
  Kind  kind = Kind::unknown;  // typecheck/type-bake
  bool  immutable = false;     // exists today
  bool  comptime  = false;
};

class Bundle {
  // NO name field — the name IS the symbol-table key (see design note below;
  // deprecated in F, deleted with 2b/H when runner_type_query_fn dies)
  Mode        mode;            // mut/const/reg — property of the BINDING, not each leaf
  std::string type_name;       // nominal identity for `is`; empty = anonymous (SSO, no alloc)
  Entry       root;            // scalar fast path — zero map nodes for scalars
  std::map<std::string, Entry, Canonical_less> data;   // tuple leaves only
  std::map<std::string, Const, Canonical_less> attrs;  // residual attrs, bare names
};

Both maps stay plain inline members (decided 2026-06-09): an empty std::map does not heap-allocate — the tree header lives inline in the object and nodes are allocated per entry — so the scalar-only case (the common one: just root) pays ~96 B of header inside the single make_shared<Bundle> block and zero extra allocations. A unique_ptr-lazy wrapper would save ~80 B per bundle but add a first-use allocation plus an indirection and null-check in every accessor — not worth it at symbol-table scale; the real per-node map-allocation cost on actual tuples is the btree/SBO follow-up below.

Sizing: Const is Dlop, a value type with inline small-value storage (core/const.hpp) — four extra Consts cost roughly +70–100 B per leaf. Acceptable because scalars dominate (they pay it once, inline) and tuples are small. Canonical_less is kept and load-bearing on the data map — numeric unnamed ordering (2 before 10) plus contiguous prefix runs for lower_bound. Using the same comparator on attrs keeps a field's attr run aligned. The flatten-to-dotted-leaf-keys compare contract is unchanged for tuples: two bundles are equal iff their flattened leaf-path→value maps are equal; the inline root only changes where a scalar's single leaf lives, not the comparison semantics (compare_bundles_eq/structural_does/Verilog naming ride on it).

Design notes

The __ alias bug this fixes. Because attrs and data share one keyspace, a data field literally named __x lands in the attribute namespace, and the detector special-cases ___ (triple underscore) to avoid false positives. With a separate attrs map the namespaces no longer overlap. The Pyrope-facing rule already says attributes have no __ prefix; this brings storage in line with the contract.

Attribute names actually in flight (verified 2026-06-09): enumentry (constprop.cpp:114–119,1933), pub/pub_unit (constprop.cpp:2862,2866), dp_assign + bits/max/min/ubits/sbits (only in the two external suffix-matchers — and bits/max/min become typed fields in D, while ubits/sbits are stale and get pruned), plus the category-B wiring names registered in upass_attributes_wiring.cpp. The residual attrs map is genuinely small.

Why the data map stays flat (and does not recurse). The O(N)→O(log N + k) sub-bundle access comes from lower_bound on the sorted flat map, and zero-copy iteration comes from walking the bounded range. A recursive node would re-introduce the category/numeric comparator at every level and fork the flatten-compare contract that compare_bundles_eq / does / Verilog naming depend on. The dotted key encodes the full path losslessly.

lower_bound boundary caveat. The stop condition is a segment boundary, not a string prefix: a bounds a.b but not ab. Reuse match_first_partial (it already encodes the .-boundary rule); lower_bound only finds the start of the run.

Storage engine follow-up (explicitly out of 1b). absl::btree_map is a near-drop-in for std::map here (ordered, lower_bound, transparent comparator) and stores small maps in one node — the 5–8-entry SBO behavior without hand-writing one. But it invalidates iterators/pointers on insert, and bundle.cpp's promotion loops erase+insert mid-walk while views hold const Entry*. Swap only after C lands, behind a bench_upass.sh measurement, with the invalidation audit done.

name field retirement. A Bundle's name is the symbol-table key; storing it again on the Bundle is replication. Verified 2026-06-09: the only functional reader is constprop's resolve_field_operand (upass_constprop.cpp:1744), which builds "<name>.<field>" to query runner_type_query_fn — exactly the pull plumbing 2b/H deletes (and the typed per-field Entry facts from D remove the need even earlier). Everything else is cosmetic: dump() (bundle.cpp:1023), two diagnostic prints (:548, :905), and concat deriving a sub-bundle name (:514). Diagnostics still need a variable name, but the dispatcher owns it: the 2b runner resolved the operand from the table, so it passes the dst/src names to the hooks as string_views — tmps/make_const wrappers never had a meaningful name anyway, and dump() grows a name parameter. Sequencing: deprecate here (F), delete in 2b/H. Bonus: dropping the const std::string member makes Bundle copy-assignable, simplifying the COW clone path (symbol_table.cpp:83).

Behavior that must round-trip exactly (LEC-pinned corners; most get simpler once attrs are split and the root is inline):

  • Scalar carriers with attrs. add_int copies attribute leaves under a scalar key so get_bundle(key) round-trips the parse-time enumentry tag. New shape: root Entry carries the value; attrs has enumentry — the scalar fast path can no longer mistake an attr leaf for a data leaf.
  • Scalar→.0 promotion. Becomes the root→data spill in E; the mid-walk is_attribute special-casing (bundle.cpp:654,662–665) drops out because attrs are no longer interleaved.
  • concat merge rules. attrs = rhs-wins; named = per-name merge with nil-drop and scalar-pack into a growing sub-tuple; unnamed = append at the next free index. Preserve precisely — pinned by the ++ / bundle-literal tests.
  • Single-element collapse. (c=3) == 3 stays a Pyrope rule in upass_constprop.cpp, not in Bundle — it is not flat-key-set equality and must not migrate into storage.
