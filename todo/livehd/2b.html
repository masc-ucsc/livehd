2b upass push-based dispatch — one runner-owned scope-aware symbol table; the runner resolves operands to Bundles and pushes (dst, src…) into each pass

Today every value-producing upass keeps its own per-variable-name side map and re-resolves operands by walking the cursor: uPass_constprop::st (Symbol_table), uPass_typecheck::kind_map, uPass_attributes's ~12 maps (type_info_map, attr_set_values, tuple_shapes, range_bounds, the alias maps, …), uPass_bitwidth::range_map_, uPass_func_extract's closure-capture maps. Each pass independently answers "what is ref foo?" from a different flat name → fact map, and the per-node hooks (process_plus() …) are nullary — they re-walk the children and look the operands up themselves. Two consequences:

  • Replication. The same per-name facts (value, kind, declared range, derived bitwidth range, cycle range, mode, comptime, sticky) live in 4–5 parallel maps, all keyed by the same name.
  • No scope. The flat maps don't track lexical scope. Demonstrated bug: uPass_typecheck::kind_map conflates same-named variables in sibling scopes — two if arms each declaring a block-local t of a different kind raise a spurious assign-type-mismatch (the no-shadowing rule only bans enclosing reuse, and SSA is a no-op on the top-level module, so the bare names collide). See Design notes.

2b flips the model. There is one runner-owned, scope-aware Symbol_table holding exactly one shared_ptr<Bundle> per live name. The runner resolves each operand of a node to a Bundle and pushes them into each pass:

Vote process_<op>(Bundle& dst, std::span<const std::shared_ptr<const Bundle>> src);
//   ^ live symbol-table bundle      ^ child-ordered operands; a `const` literal is a
//     for the dst name                throwaway trivial Bundle owned by the iterator

Passes stop owning symbol tables; they read the pushed-in src bundles and write their slice of dst. The per-name facts move onto the Bundle as first-class typed attributes (delivered by 1b). This retires ~26 maps and the pull-query plumbing, and the single scope-aware table makes the sibling-scope bug structurally impossible.

Depends on

1b — hard prerequisite. 2b consumes 1b subtasks E–J (the Bundle API surface): the shared enum class Kind (lnast/kind.hpp) + set_kind/get_kind; the typed set_attr/get_attr_int|bool|str overloads; scalar() / has_scalar(); the battr:: attr-key vocabulary; the per-attribute sticky bit + sticky_attributes() view; and the cheap trivial bundle + make_const(Const, Kind) factory. Do not start 2b until 1b is green.

Flag-day. Per the migration decision: convert the runner + all value-producing passes in one change; the old nullary-hook / private-map path is removed in the same change. Other upass work freezes until 2b lands. The diff is large and not bisectable by design — lean on the full suite + LEC pairs (subtask I) as the gate.

Target shape

  • One runner-owned scope-aware symbol table. Resurrect the runner_symbol_table seam (deleted as dead scaffolding) — wired this time. One shared_ptr<Bundle> per name, with block_scope push/pop so sibling scopes do not alias. The previous node's fully-populated dst is the next node's src; all cross-pass information flows through the bundle, not through pull callbacks.
  • Hook signature. Vote process_<op>(Bundle& dst, std::span<const std::shared_ptr<const Bundle>> src) for the value-producing ops, where Vote = keep | drop. src is in child order (order matters: minus/div/mod/shl/sra/ordering-compares/get_mask/sext/tuple_get/range). A const operand is a throwaway trivial Bundle (make_const, iterator-owned, valid for the hook call only); a ref operand is the name's live bundle from the table.
  • Emit model (a) — bundle is the single source of truth. Passes mutate dst and return only keep/drop. After dispatch the runner derives the opt node: if dst.scalar() is comptime-known → emit assign(dst, const); else emit the op with operands folded from src; drop elides it. Structural rewrites (inline / specialize / unroll) are replace and stay runner-owned, not a pass vote.
  • Slice ownership. Metadata attrs are single-owner / disjoint (one writer each) → pass dispatch order is irrelevant. The value slice is shared, must-agree: more than one pass may pin a constant (constprop, and bitwidth via a single-point range); if two pin, the values must be identical — a mismatch is an I() internal compiler error (a LiveHD bug, not a user diagnostic). An abstaining pass (leaves it unset) never conflicts with a pinner. Any order yields the same constant or aborts.
  • dst is the live bundle; two slice lifetimes. Only a declaration creates+inserts a bundle; every assignment mutates that same entry. So mut a:u3=nil; a=3; a=b+2 — all three touch one a bundle. Persistent slices (type/Kind, mode, declared max/min) are set once at the declaration and survive every assignment (typecheck's no-type-change check reads dst.get_kind() straight off the live bundle). Per-write slices (the value scalar, derived bw_max/bw_min) are reset by the runner at each store before dispatch so the stale prior value cannot leak, then re-pinned for this definition.
  • Declaration pre-step bakes the type subtree once. The declare/type_spec TYPE subtree (prim_type_int(max,min), named type, comp_type) is read once by the runner at the declaration and baked into the bundle's persistent attrs (type via Kind, declared max/min, mode). After that no pass ever walks a type subtree — they read the attrs. Today that subtree is re-walked by ~4 passes (attributes, bitwidth, constprop, typecheck).
  • function_scope read-through / write-block. The one new Symbol_table behavior. Today find_decl_scope (lnast/symbol_table.cpp:24–33) treats a Function scope as a hard barrier — a callee body cannot reach even read an outer decl. Change it so a read crosses the function barrier (closure capture of outer comptime consts) but a write to a name that resolves only above the barrier is a compile error (the no-side-effect rule prp2lnast already enforces lexically). This is what lets func_extract drop its closure-capture maps — the body, walked under function_scope on the shared table, reads outer consts directly and drops its locals on leave_scope.
  • func_call via a runner-owned Call_resolver helper. Keep the runner lean — do not pile call handling into uPass_runner. A Call_resolver absorbs the call/return surface (see subtask F). Resolved-call opt-tree shape:
    • args in = one arg tuple baked into the opt call, fully named to the callee params — except a vararg callee (...x), where surplus actuals bind positionally into the vararg sub-tuple (unnamed allowed only there). A vararg callee is specialized per signature into its own LNAST (add1__v1, add1__v2, … one per distinct vararg arity/type — the existing 1p path), so a Sub that reaches tolg is always a concrete, fully-named signature.
    • result out = a single bundle (one tuple; named and/or unnamed fields). Consumed via the existing bundle-to-bundle Named-tuple destructuring rules (docs/pyrope/03-bundle.md §Named-tuple destructuring): named result → bind by name (bare b ← field b; missing field = compile error; local = src.path for rename/nested; LHS order irrelevant); unnamed result → positional. The result is just a bundle RHS — reuse the destructuring path, no special "call" handling.
    • tolg wires by name (everything named in the lgraph), never by position.
    • scope: this treatment applies to calls that survive to tolg — pipe/mod Sub (and import → namespace bundle on the dest). Builtins (cputs/cassert/assume) keep their bespoke positional shape (consumed by verifier/assert/typecheck, never reaching tolg as calls). Comb calls are inlined away — no call node.
  • Overflow = error at the node. bitwidth emits a does-not-fit overflow through the diag sink at the offending store (like typecheck) — an error-severity diag already fails the compile non-zero, so there is no end_run throw and no deferral. bitwidth loses write_site_ and pending_overflow_msg_.

Design notes

The sibling-scope bug this fixes. With one flat kind_map and no scope pop, if c { mut t = 1; o = t } else { mut t = true; o = int(t) } raises a spurious upass.typecheck: cannot assign boolean value to 't' (it is integer) — the else-arm's fresh t sees the then-arm's kind. Same-kind siblings compile clean, proving it is pure flat-map conflation, not a scope rule. The runner-owned scope-aware table (with block_scope push/pop) resolves each arm's t to its own bundle, so the bug cannot occur.

Why the value slice is must-agree, not single-owner. Both constprop and bitwidth can legitimately conclude a constant (constprop by folding, bitwidth by deriving a single-point range). They must reach the same constant; if they don't, it is an internal compiler bug, not a user error — hence I(), and hence order-independence is preserved.

Why flag-day. The hook signature change touches every value-producing pass and the runner's dispatch/emit core at once; a coexistence mode would mean dual-dispatch scaffolding (push-based and nullary-cursor hooks) for the duration. Cleaner to convert together and gate on the full suite + LEC. Other upass work freezes until it lands.

func_call taxonomy (why the passes touch func_call today, and where it goes): A — intrinsics (__sum/casts/wrap/sat) are ops in a call costume → they flow through the normal (dst, src) hook (or a name-keyed fold path), not special call handling. B — builtins (cputs/cassert/assume) are side-effecting / checking, no value dst → stay as their own small handlers. C — imports → the Call_resolver binds the external unit onto the dest bundle. User-function arg binding + inline/specialize is already the runner's job; the Call_resolver just gives it a home.

Out of scope. The tolg / Time_checker graph domain is untouched (it is a separate pass over the post-upass tree). timecheck stays a standalone post-walk pass; 2b only standardizes its cycle interval onto the bundle as tc_* so it reads inputs uniformly — folding the pass into the per-node dispatch is a possible later step, not 2b.
