2b upass push-based dispatch — one runner-owned scope-aware symbol table; the runner resolves operands to Bundles and pushes (dst, src…) into each pass

Today every value-producing upass keeps its own per-variable-name side map and re-resolves operands by walking the cursor: uPass_constprop::st (Symbol_table), uPass_typecheck::kind_map, uPass_attributes's ~12 maps (type_info_map, attr_set_values, tuple_shapes, range_bounds, the alias maps, …), uPass_bitwidth::range_map_, uPass_func_extract's closure-capture maps. Each pass independently answers "what is ref foo?" from a different flat name → fact map, and the per-node hooks (process_plus() …) are nullary — they re-walk the children and look the operands up themselves. Two consequences:

  • Replication. The same per-name facts (value, kind, declared range, derived bitwidth range, cycle range, mode, comptime, sticky) live in 4–5 parallel maps, all keyed by the same name.
  • No scope. The flat maps don't track lexical scope. Demonstrated bug: uPass_typecheck::kind_map conflates same-named variables in sibling scopes — two if arms each declaring a block-local t of a different kind raise a spurious assign-type-mismatch (the no-shadowing rule only bans enclosing reuse, and SSA is a no-op on the top-level module, so the bare names collide). See Design notes.

2b flips the model. There is one runner-owned, scope-aware Symbol_table holding exactly one shared_ptr<Bundle> per live name. The runner resolves each operand of a node to a Bundle and pushes them into each pass:

Vote process_<op>(Bundle& dst, std::span<const std::shared_ptr<const Bundle>> src);
//   ^ live symbol-table bundle      ^ child-ordered operands; a `const` literal is a
//     for the dst name                throwaway trivial Bundle owned by the iterator

Passes stop owning symbol tables; they read the pushed-in src bundles and write their slice of dst. The per-name facts move onto the Bundle as first-class typed fields (delivered by the landed 1b refactor). This retires ~26 maps and the pull-query plumbing, and the single scope-aware table makes the sibling-scope bug structurally impossible.

Target shape

  • One runner-owned scope-aware symbol table. Resurrect the runner_symbol_table seam (deleted as dead scaffolding) — wired this time. One shared_ptr<Bundle> per name, with block_scope push/pop so sibling scopes do not alias. The previous node's fully-populated dst is the next node's src; all cross-pass information flows through the bundle, not through pull callbacks. Scope/if-arm contract (decided 2026-06-09): keep today's semantics. Every { } notifies the table (scope enter/exit); a declaration inside an arm allocates its entry in that scope and the entry is destroyed at }; a write to a name declared in an enclosing scope under a runtime-uncertain condition invalidates that name's value fact on arm exit (today's mark-uncertain + invalidate-on-exit, symbol_table.cpp:282–357). No frame-per-arm merge: the both-arms-write-the-same-constant fold is deliberately not attempted.
  • Hook signature. Vote process_<op>(Bundle& dst, std::span<const std::shared_ptr<const Bundle>> src) for the value-producing ops and for store, where Vote = keep | drop. src is in child order (order matters: minus/div/mod/shl/sra/ordering-compares/get_mask/sext/tuple_get/range). A const operand is a throwaway trivial Bundle (make_const, iterator-owned, valid for the hook call only); a ref operand is the name's live bundle from the table. Bundles carry no name — the name is the symbol-table key, and the runner (which just resolved each operand from the table) supplies the dst/src names to the hooks as string_views for diagnostics; Bundle::name is deleted in subtask H together with its last functional reader (constprop's resolve_field_operand building dotted keys for runner_type_query_fn). store is not just runner-side plumbing: it still dispatches to the passes so typecheck, bitwidth, attributes, and constprop can validate and update dst. For a field-path store, dst is the whole destination bundle (for example t in store(t,b,tmp)), and the path selectors plus value operand are passed in src in LNAST child order.
  • Emit model (a) — bundle is the single source of truth. Passes mutate dst and return only keep/drop. After dispatch the runner derives the opt node: if dst.scalar() is comptime-known → emit a constant store; else emit the op with operands folded from src; drop elides it. Structural rewrites (inline / specialize / unroll) are replace and stay runner-owned, not a pass vote.
  • Slice ownership. Metadata attrs are single-owner / disjoint (one writer each) → pass dispatch order is irrelevant for those independent slices, subject to the dependency graph in upass/README.md. The value slice is shared, must-agree: more than one pass may pin a constant (constprop, and bitwidth via a single-point range); if two pin, the values must be identical — a mismatch is an I() internal compiler error (a LiveHD bug, not a user diagnostic). An abstaining pass (leaves it unset) never conflicts with a pinner. Any order yields the same constant or aborts.
  • dst is the live bundle; copy-on-write for self-use. A declaration creates+inserts the first bundle for a name. An assignment normally mutates the current live bundle for that name, preserving the declaration-persistent slices (type/Kind, mode, declared max/min) so typecheck's no-type-change check reads dst.get_kind() straight off the bundle. The repeated user-variable store store(c,tmp) mutates c's existing bundle in place after tmp has been resolved; it does not allocate a fresh replacement bundle. The runner must not clear value until after the RHS has been resolved: source-level assignments such as mut a=3; a=a+1 are already split into producer nodes (for example plus(tmp,a,1)) followed by a store(a,tmp), and the producer must see the old a value. Current LNAST has no assign or tuple_set node; the single mutating write node is store, and it still calls the passes. A scalar-to-scalar store clears the whole destination value before applying the new value. A tuple-to-tuple store(c,tmp) is a merge/update, not a tuple wipe: if c already has fields, tmp may assign a subset of those fields, and fields absent from tmp remain known/preserved. It is a typecheck error for a scalar destination to receive a tuple RHS, for a tuple destination to receive a scalar RHS, or for a tuple RHS to introduce fields not present in the destination shape. A field-path store clears only the written entry/path: mut t=(a=1,b=2); t.b=runtime_input lowers to a store on path b, so t.a==1 stays known while t.b becomes unknown until a pass re-pins it. Reads of fields are likewise already explicit tuple_get producer nodes, so t.b=t.a+1 is seen as tuple_get(tmp0,t,a), plus(tmp1,tmp0,1), store(t,b,tmp1); t.a and t.b are separate Entrys in the Bundle's data map (upass/core/bundle.hpp) and do not require copying. Copy/new-entry handling is only needed if a normalized store's value operand is the same exact entry being cleared. A direct self-store such as store(c,c) (c = c) is a user-facing syntax diagnostic rather than a no-op or a copy/clear special case: message irrelevant assignment, hint likely an error or delete this assignment. This diagnostic is deliberately narrow: only the obvious direct lowered form is checked; do not add path-equivalence analysis for cases such as source-level c.a = c.a. Metadata that must not change across assignments remains declaration-persistent.
  • store path semantics. process_store receives the destination aggregate as dst, not a detached sub-bundle. The intermediate store children are selectors, and the last child is the value being stored. The runner resolves every selector to a bundle before dispatch: a literal selector such as const 'a' becomes a fake one-entry const bundle containing a, and a ref selector becomes whatever bundle that ref resolves to. Passes do not inspect whether the original child was const or ref; they consume the selector bundle's const value when available. Passes use those resolved selectors to update/check the destination entry: typecheck verifies the selected field's type is compatible with the stored value type, and bitwidth copies/derives the selected field's bw_max/bw_min from the stored value's range. Attribute facts on the destination bundle and its subfields are preserved across stores. If the RHS carries an attribute for a destination entry that already has a fixed attribute, the store is a compile error rather than silently overwriting or unsetting the attribute. A store(c,___3) from an immutable tmp mostly copies compatible facts from ___3 into c or the selected field, but only after typecheck validates compatibility. If the destination entry has no type yet, the store copies/binds a basic scalar RHS type into that entry; later stores to the same entry must match that inferred type. Example: mut c=nil; c=3; c="foo" errors on the string store because c=3 bound c to integer. nil does not mean "infer any aggregate shape later": a store such as mut c=nil; c=(a=3) is a typecheck error because it tries to assign a tuple to a destination waiting for a simple type. Adding fields later is likewise not allowed unless the destination already has a declared/inferred tuple shape that contains those fields. When the destination already has tuple shape, inference is per field: c.a=3 can bind only field a to integer while c.b remains unset; a later c.a="foo" is an error. If an unset/nil field is read or reaches LGraph generation as a used value, emit an error. If it remains unset but is never used, emit a warning rather than a hard compile failure. For arithmetic producers such as plus(___3,...), the producer pass has already made ___3 an integer-like bundle with its own value and/or bw_* facts; store can copy those facts to the destination. Missing RHS attrs never delete destination attrs. All attrs are add-only once present: setting the same value again is a no-op, and setting a different value emits a user-facing diagnostic that attributes with different values cannot be changed once set. The canonical leading-_ name controls propagation, not mutability: sticky (_-named) attrs are copied through stores, while non-sticky attrs are not copied by store(a,b) and remain local to their existing bundle/entry. The attributes upass owns the user-facing diagnostic for add-only attr conflicts; other passes should not emit their own competing attr-conflict diagnostics.
  • Declaration pre-step and standalone type_spec bake type subtrees once. The declare type child and any standalone type_spec(dst,type) subtree (prim_type_int(max,min), named type, comp_type) are read once by the runner and baked into the destination bundle's persistent attrs (type via Kind, declared max/min, mode when applicable). Standalone type_spec(___3,...) creates/updates the ___3 bundle in the shared symbol table like any other producer, so later tuple/build/store logic can carry those attrs onto fields. This is not special to type_spec: every producer node with a compiler tmp destination, such as plus(___3,...), writes that tmp exactly once. The runner tracks the tmp in the symbol table and marks the resulting bundle immutable after that producer's dispatch finishes. Later LNAST nodes may read the tmp but may not add metadata, values, or attrs to it; all facts for a tmp must be written by the passes dispatched on the producer node that creates it. If any later LNAST node tries to use the same tmp as a destination again, or tries to mutate the immutable tmp bundle, that is an internal compiler error. After baking, no pass ever walks a type subtree — they read the attrs. Today that subtree is re-walked by ~4 passes (attributes, bitwidth, constprop, typecheck).
  • function_scope read-through / write-block. The one new Symbol_table behavior. Today find_decl_scope (upass/core/symbol_table.cpp:20–36) treats a Function scope as a hard barrier — a callee body cannot reach even read an outer decl. Change it so a read crosses the function barrier (closure capture of outer comptime consts) but a write to a name that resolves only above the barrier is a compile error (the no-side-effect rule prp2lnast already enforces lexically). This is what lets func_extract drop its closure-capture maps — the body, walked under function_scope on the shared table, reads outer consts directly and drops its locals on leave_scope.
  • func_call via a runner-owned Call_resolver helper. Keep the runner lean — do not pile call handling into uPass_runner. A Call_resolver absorbs the call/return surface (see subtask F). Resolved-call opt-tree shape:
    • args in = one arg tuple baked into the opt call, fully named to the callee params — except a vararg callee (...x), where surplus actuals bind positionally into the vararg sub-tuple (unnamed allowed only there). A vararg callee is specialized per signature into its own LNAST (add1__v1, add1__v2, … one per distinct vararg arity/type — the existing 1p path), so a Sub that reaches tolg is always a concrete, fully-named signature.
    • result out = a single bundle (one tuple; named and/or unnamed fields). Consumed via the existing bundle-to-bundle Named-tuple destructuring rules (docs/pyrope/03-bundle.md §Named-tuple destructuring): named result → bind by name (bare b ← field b; missing field = compile error; local = src.path for rename/nested; LHS order irrelevant); unnamed result → positional. The result is just a bundle RHS — reuse the destructuring path, no special "call" handling.
    • tolg wires by name (everything named in the lgraph), never by position.
    • scope: this treatment applies to calls that survive to tolg — pipe/mod Sub (and import → namespace bundle on the dest). Builtins (cputs/cassert/assume) keep their bespoke positional shape (consumed by verifier/assert/typecheck, never reaching tolg as calls). Comb calls are inlined away — no call node.
  • Overflow = error at the node. bitwidth emits a does-not-fit overflow through the diag sink at the offending store (like typecheck) — an error-severity diag already fails the compile non-zero, so there is no end_run throw and no deferral. bitwidth loses write_site_ and pending_overflow_msg_.

Design notes

The sibling-scope bug this fixes. With one flat kind_map and no scope pop, if c { mut t = 1; o = t } else { mut t = true; o = int(t) } raises a spurious upass.typecheck: cannot assign boolean value to 't' (it is integer) — the else-arm's fresh t sees the then-arm's kind. Same-kind siblings compile clean, proving it is pure flat-map conflation, not a scope rule. The runner-owned scope-aware table (with block_scope push/pop) resolves each arm's t to its own bundle, so the bug cannot occur.

Why the value slice is must-agree, not single-owner. Both constprop and bitwidth can legitimately conclude a constant (constprop by folding, bitwidth by deriving a single-point range). They must reach the same constant; if they don't, it is an internal compiler bug, not a user error — hence I(), and hence order-independence is preserved.

Why flag-day. The hook signature change touches every value-producing pass and the runner's dispatch/emit core at once; a coexistence mode would mean dual-dispatch scaffolding (push-based and nullary-cursor hooks) for the duration. Cleaner to convert together and gate on the full suite + LEC. Other upass work freezes until it lands.

func_call taxonomy (why the passes touch func_call today, and where it goes): A — intrinsics (__sum/casts/wrap/sat) are ops in a call costume → they flow through the normal (dst, src) hook (or a name-keyed fold path), not special call handling. B — builtins (cputs/cassert/assume) are side-effecting / checking, no value dst → stay as their own small handlers. C — imports → the Call_resolver binds the external unit onto the dest bundle. User-function arg binding + inline/specialize is already the runner's job; the Call_resolver just gives it a home.

Out of scope. The tolg / Time_checker graph domain is untouched (it is a separate pass over the post-upass tree). timecheck stays a standalone post-walk pass and does not write normal Bundle facts. If it needs propagation state, that state is timecheck-owned; folding the pass into the per-node dispatch is a possible later step, not 2b.
