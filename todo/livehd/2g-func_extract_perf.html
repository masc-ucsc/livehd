2g func_extract is the slowest upass — move extraction to the lnast creators

func_extract is the single most expensive pass.upass sub-pass, even on inputs that define no comb/pipe/mod lambdas. Goal: stop running it as a post-hoc, re-walk-and-re-materialize pass, and instead have the lnast creators (inou/prp and inou/slang) emit the multiple LNASTs at creation time, turning each function definition site into a plain import reference. The dedicated func_extract pass is then deleted.

Why it is expensive today

Unlike every other sub-pass, func_extract does not ride the single main-pipeline walk. pass_upass.cpp runs it as a dedicated pre-loop: for every entry-point LNAST it builds a fresh uPass_runner over extract_order = {"func_extract"} and calls run() (pass/upass/pass_upass.cpp:383–410). Three compounding costs, all paid even when there is nothing to extract:

  • 1. Separate full traversal. An extra DFS of every top-level LNAST before the main pipeline even starts.
  • 2. Unconditional re-materialization (dominant term). The pre-loop runner is created without set_materialize(false), so materialize_ defaults to true (upass_runner.hpp:57,113). It builds a complete staging copy of the module's tree via emit_* and the caller replace_body()s it in — even when zero func_defs were found and nothing was dropped.
  • 3. Per-node closure-capture bookkeeping. Overrides of process_assign/store/plus/…/tuple_add/ func_call fire on every matching node with save_cursor/restore_cursor + unordered_map churn (latest_outer_value, temp_scalar_value, fold_temp_nary, …), all only to inline outer-scope constants into extracted bodies — wasted with no func_defs.

The pass was bolted onto the runner as the path of least resistance; the runner already reserves the seam for retiring it — function_registry, "the function-body registry the runner owns once func_extract collapses into the main walk" (upass_runner.hpp:90–96, "Step L of the upass redesign"). This task goes one step further than Step L: push the split all the way up to the lnast creators.

What func_extract actually does (the behavior to preserve)

Beyond dropping each func_def node and registering the body, the pass does exactly four things — none of which need a separate post-walk:

  • Structural split. Mint a top → [io, stmts] LNAST per comb/pipe/mod def, copy the signature verbatim, stamp lambda kind / lg name / SourceId anchor.
  • Template flag. set_template(true) when the signature is not fully concrete: an untyped non-self input, a ... varargs default, or any generic (stamp_template_if_untyped, upass_func_extract.cpp:71–111). Everything downstream gates on is_template() — tolg/timecheck/bitwidth skip templates (tolg.cpp:4400,4414; bitwidth.cpp:338,365); the main walk runs templates non-materializing and drops their standalone clones (pass_upass.cpp:449,467,488).
  • Generic names. set_generics(<T,U>) verbatim — a seam only; per-T substitution is todo 3g (deferred).
  • Closure capture. Inline the outer-scope comptime constants the body reads (const K = 42 → prelude store K = 42) so the extracted body is self-contained. Done via a partial Dlop fold during the walk (fold_temp_nary). This is the only non-syntactic piece.

The real specialization is not here — it is at call sites in the runner and stays untouched: a comb template inlines (constprop folds per call, so each site can specialize differently); a pipe/mod/fluid template is cloned by make_specialized_clone (runner.cpp:~2880–2967) which does set_template(false), set_generics({}), and injects the concrete port types from the resolved actuals; generic binds resolve once via resolve_generic_binds (runner.cpp:2991+). Clones dedup by mangled signature.

Proposed design

The lnast creators already know where every function is — comb/pipe/mod name(...) is an explicit grammar production in prp2lnast, and slang knows its module/function boundaries. So the split is purely syntactic and belongs in the front-end:

  • Front-end emits one LNAST per comb/pipe/mod at creation, with is_template, generics, lambda kind, lg name and the SourceId anchor all stamped directly from the AST (no re-deriving by walking a copied io). The runner needs a callee body to inline/specialize from regardless of kind, so a tree is always produced.
  • The definition site becomes a reference: const fun1 = import("ln:localfile.fun1") — binds the name to the freshly-minted sibling LNAST. This makes the function dependency first-class in the IR (helps the LSP, incremental rebuild, and the depfile / import-resolver feed of 3y-agent). The resolver scheme already exists — call_resolver.cpp:174,188 handle ln: (bind to the lnast tree name, resolved against the function registry; the inline-able form, already used as const f = import("ln:u.g") at runner.cpp:1830) and lg: (bind the url, lowered to a Sub instance at tolg). Bare import("file.entity") remains the cross-file pub-entry form (:199+). So nothing new is needed in the resolver; the front-end just emits the ref and registers the minted tree. pub just rides the same ref: pub const fun1 = import("ln:localfile.fun1") re-exports; drop pub for file-local. No special func-registry list.
  • The dedicated func_extract pass and its pre-loop are deleted. The runner's template inline/specialize machinery is unchanged.

Unit-vs-inline is an optimization, not a correctness gate

Inlining is always legal. For comb it is a splice; for mod/pipe it is hierarchy flattening (equivalent hardware — each call site gets its own logic+state copy whether realized as a Sub instance or inlined; pipe flop stages travel with the body). The only cost of over-inlining is code/area bloat, and locally-defined functions are inlined most of the time anyway. Consequences:

  • Whether a tree is emitted as a standalone lowerable unit (its own LGraph module + Sub instances) versus a call-site-only callee is a bloat/hierarchy knob — and it is expressed directly by which import scheme the def site emits: import("ln:…") exposes the lnast tree so the runner can inline it, while import("lg:…") forces a Sub instance at tolg (preserves the module boundary). Default: emit lg: for pub/top mod/pipe (preserve the boundary for instancing, synthesis hierarchy, debug, and to avoid duplicating big/multiply-instantiated bodies); emit ln: and inline everything else. Note tolg already lowers a concrete non-template tree standalone and skips templates (tolg.cpp:4400) — keep that, just feed it front-end-built trees.
  • Every hard edge has a safe fallback: inline it. Forward reference we could not pre-resolve, ambiguous typed-vs-template, a callee whose interface is not ready, or a fiddly cross-unit capture — none can produce wrong hardware, so each degrades to "a bit more code," never a blocker.

Correctness condition — resolve callee interfaces independent of tree order

Today this is free: a global SSA pre-loop harvests io_meta into every tree before the main walk (pass_upass.cpp:422–426), so any callee's interface is known regardless of definition order, forward refs, or mutual recursion. Front-end emission must preserve that guarantee. Two options:

  • (a) Front-end populates io_meta directly — chosen. prp/slang already compute the full signature (param names, types/ranges, output names, stages annotations) to decide typed-vs-template, so filling io_meta is nearly free and makes every callee interface queryable independent of processing order — forward refs, mutual recursion, and cross-file import("ln:…") (which have no single serial order) all just work. It subsumes the io-harvest half of the SSA pre-pass (body SSA-renaming still runs; its io-harvest becomes a no-op when io_meta is pre-filled). Partial io_meta (a param typed by a comptime expr like int(MAX)) is fine — the unknown range resolves later in bitwidth / at specialization, which coincides with the template flag anyway.
  • (b) Enforce serial / topological tree order — rejected. Fragile against forward references and mutual recursion, and undefined across imported files. Trades a cheap data-population step for a brittle scheduling constraint.

Closure capture is orthogonal to typed-ness

Even a fully-typed comb can close over an outer comptime constant, so concrete trees need capture too — it is not a template-only concern. Replace the partial fold_temp_nary with a capture list recorded by the front-end (the free variables the body reads that are bound in the enclosing scope), resolved with full constprop fidelity when the defining unit is processed (where the def-scope consts are live). This matters for lexical scoping: a pub function imported and called from another file closes over its definition scope, not the caller's — so the capture must be baked at the defining unit before any foreign call site sees it, never deferred to the call site. For file-local functions (the common case) the captured consts are in the same scope and trivially available.

Depends on

None. Touches inou/prp (prp2lnast), inou/slang, pass/upass/pass_upass.cpp, and lnast/ (io_meta + flags at creation), and removes upass/func_extract/. The ln:/lg: import schemes already exist (call_resolver.cpp:174,188) — no resolver change. Coordinates with 3g (generics seam) and 3y-agent (import-ref dependency feed).

Interim mitigations (if the redesign is staged)

Cheap wins that reduce the cost without the front-end rework, usable as stepping stones:

  • Cheap pre-scan + skip. Before the pre-loop runner, scan (or reuse a parse-time flag) for any comb/pipe/mod func_def; if none, skip the whole iteration — no walk, no staging, no replace_body. Zeroes the no-functions case.
  • Skip re-materialization when nothing was dropped. Run the pre-loop non-materializing and only build/swap a staging tree on modules that actually extracted a def. The replace_body full-tree swap is the bulk of the cost.

Subtasks — landed

2026-06-15 — the perf core (B/C/D/F) landed. The runner-based func_extract upass is deleted; the comb/pipe/mod lambda split is now a direct, materialize-free tree surgery (upass::extract_lambda_functions in upass/func_extract/upass_func_extract.cpp) run once per entry-point tree at the pass.upass pipeline entry (replacing the old pre-loop in pass_upass.cpp). It emits the SAME per-function top → [io, stmts] trees the old pass did (signature verbatim, lambda_kind/lg/generics/ SourceId stamped, is_template for not-fully-typed sigs, the outer-scope comptime closure-prelude at the body head), then drops the func_defs from the module body via one replace_body rebuild — only when a lambda was actually present. On the function-free benchmark (xx.prp, 100k stmts) it is now a cheap pre-scan that does nothing; the old pass paid a full re-materialization regardless.

  • Measured (F). bench_upass.sh on xx.prp: full pipeline 0.652s → 0.544s (the .122s func_extract row is gone — it is no longer a pass); LSP one-shot 0.570s → 0.436s. No prp/slang/lhd/LEC regressions (//inou/prp:all 484/484, //inou/slang:all 118/118, //lhd/tests 39/39).
  • Where it differs from the plan above. The split runs at the pipeline ENTRY, not literally inside prp2lnast/slang. That boundary is what covers every compile path (the kernel compile flow AND the LSP both call Pass_upass::work) while leaving the parse-only dumps untouched (ln.cat/scan/prp-writer never run upass), and it preserves the is_function_body positional semantics (idx ≥ original_lnast_count) byte-for-byte. slang already emits the extracted unit form, so it has no func_def to split and is unaffected (confirming the plan's slang note).
  • Deferred (intentionally, as the plan allows). A (io_meta at creation) — the SSA pre-loop still harvests io_meta from the emitted io node, which is sufficient and avoids a double-population guard. B's import-ref form (const name = import("ln:file.name")) and E (the ln: vs lg: unit-vs-inline policy) — the func_def is simply dropped as before and calls resolve through the name-keyed function registry exactly as today, so no new declared-once/shadowing surface and no forced-Sub hierarchy change (which would have shifted LEC golden hardware). These are pure architectural niceties orthogonal to the perf win; pick them up with 3g/3y-agent if/when their consumers need first-class import edges.
