2opt-import Lazy --top resolution — open only the inputs XXX needs

lhd compile --top XXX file1 file2 ln:ln1 ln:ln2 … today parses and elaborates every input, then picks XXX out of the result with a pure post-elaboration name filter (filter_top, lhd_kernel.cpp:1487-1505). There is no early pruning: pyrope_parse parses all of opts.files and fully Forest::loads every ln: directory before --top is even consulted. With many ln: inputs this body-loads forests that XXX never instantiates.

2opt-import adds a cheap resolve pre-pass in front of the compile pipeline. From --top XXX it finds the single input that declares XXX (a pub decl in a .prp, or an exported unit in an ln: manifest), then walks the transitive import closure from that seed — opening an input only when it is actually reachable. Everything unreachable costs at most a manifest.json read (for ln:) or a lexer pass (for .prp), never a forest body load or a full parse. The pipeline downstream is unchanged; it just receives a pruned input set.

Mental model (one line)

Resolve --top before elaboration, not after: locate the one input that declares XXX, then chase only the imports it transitively needs — unreachable ln: forests are never body-loaded, unreachable .prp files are never parsed.

Cost model (the constraint that shapes this)

  • Parsing a .prp is acceptable. The expensive thing to avoid is body-loading ln: forests that the top never touches.
  • One ln: open is cheap, but there can be many. The cost is aggregate: eagerly Forest::load-ing 50 ln: dirs when XXX reaches 3 of them is the waste. So the design must body-load only the reachable subset.
  • The lever already on disk: every ln: directory carries a manifest.json sibling of forest.txt with a per-unit pub index (name, kind, url) and the unit names (write_manifest, lhd_kernel.cpp:721-731; populated from ln->get_pub_list() at 798-806). Reading that JSON answers "does this ln: export XXX?" with zero Forest::load. That is the cheap index this task is built on.

Depends on / relates to

  • 2i-import — its landed subtask S1 already built the lazy transitive .prp discovery this task reuses (discover_imports, lhd_kernel.cpp:1640-1759): a fixpoint that parses a seed, collects its imports, resolves each to a sibling <name>.prp, parses the new file, repeats. 2opt-import changes only what seeds it and teaches it to cross ln: boundaries. The two tasks are orthogonal: 2i-import is about link-time identity / dedup / rename; 2opt-import is about not opening inputs you don't need to reach the top.
  • lhd scan — the lexer-only import-discovery primitive (Prp2lnast::scan_imports, prp2lnast.cpp:8565-8616, Lexer(buf).tokenize() at 8570). This task adds its sibling — a lexer-only scan_pubs for declared names — and starts using both on the compile path (compile currently ignores scan_imports and walks the fully-parsed LNAST instead, via collect_imports, lhd_kernel.cpp:1961-1991).
  • The ln: persisted side-channels (io_meta / bw_meta in the manifest, lhd_kernel.cpp:650-654, 814-858) — already let a loaded mod/pipe import skip re-elaboration (set_pre_elaborated, 1787-1790). Orthogonal but adjacent: this task decides whether to load a forest at all; pre-elaboration decides how cheaply to use one once loaded.

What already exists (the 80%)

  • Lazy transitive .prp closure. discover_imports is already a fixpoint loop (seed → collect_imports → find_prp in the importer's own directory → parse the new file → repeat, lhd_kernel.cpp:1640-1759). The "follow only what you import" machinery for source is built — it is just seeded with all of opts.files (1670-1673) and skips ln:/lg: edges (continue at 1693-1695).
  • Cheap ln: export index. manifest.json lists unit names + per-unit pub entries (name/kind/url) without touching any body.bin (write_manifest, 721-731; read back in load_ln_dir at 893-907). forest.txt independently lists tree_io <idx> <name> declarations, so unit existence is cheap too.
  • Lexer-only scan. scan_imports tokenizes a .prp and returns its raw import strings without a parse or an LNAST build; import args are comptime string literals, statically extractable.
  • --top is a name filter. filter_top selects from the already-elaborated units by get_top_module_name()==XXX and throws if absent. Nothing about it prunes inputs.

What's missing (the new surface)

  • No declared-name scan for .prp. scan_imports extracts imports, not exports. There is no scan_pubs. But pub is a real lexer keyword (Keyword::kw_pub, prpparse/prp_keywords.def), so a lexer-only declared-name scan is a direct mirror of scan_imports.
  • discover_imports is seeded wide and stops at ln:. It starts from every file and never follows ln:/lg: import edges, so an ln: dependency must be listed on the CLI and is fully loaded.
  • ln: import edges are not persisted. The manifest carries names/pubs/io_meta/bw_meta but no import list. To learn what an ln: unit imports you must Forest::load its body and run collect_imports — so ln:→ln: chasing is not cheap today.
  • Forest::load is all-or-nothing. It loads every body.bin in the directory (tree_serial.cpp:512-524); there is no load-one-unit-by-name API. So once you decide you need one unit from an ln: dir, you currently pull all of that dir's bodies. (Skipping the unreachable dirs is the main win regardless; this is a finer-grained follow-on.)

Design — the resolve pre-pass

Phase 0 — build a name → provider index (no bodies, no full parse)

For each CLI input, record which top-level names it provides and what it imports, using only cheap reads:

  • ln:DIR: read only DIR/manifest.json → each pub name → (ln:DIR, unit), plus its kind / lg / url. No Forest::load.
  • .prp: run scan_pubs (new, lexer-only) → each pub name (and its lg="…" override, if any) → the .prp file; and reuse scan_imports for that file's import strings.

The result is a provider map over all inputs, built from manifests + lexer passes only — no forest bodies, no elaboration.

Phase 1 — locate the top, early and precisely

Look up XXX in the index. Not found → a fast, exact error ("top XXX is not declared by any input") before any heavy work — strictly better than today's "parse everything, then filter_top throws." Found → the single seed: one .prp file, or one ln: unit and its dir.

Phase 2 — lazy transitive closure from the seed

This is the existing discover_imports fixpoint with two edits:

  • Seed it with the one declaring input from Phase 1, not all opts.files.
  • Teach it to follow ln: edges (instead of continue): resolve each import through the Phase-0 index, and Forest::load only the ln: dirs on the path. For an ln: unit's own transitive deps, recover its imports — either from a manifest import-edge list (decision D1) or by loading its body and running collect_imports.

Phase 3 — hand the pruned set to the existing pipeline

pyrope_parse / lower_lnasts / tolg run on only the reached files and ln: units. Unreached inputs were never parsed or body-loaded. The rest of compile is untouched; filter_top still selects XXX at the end (now from a smaller set).

Decisions to settle

  • D1 — persist ln: import edges, or full-load each touched dir? To chase an ln: unit's transitive deps you need its imports, which are not in the manifest today. Option A: accept a full Forest::load per touched ln: dir + collect_imports (still skips all unreachable dirs — the main win). Option B (recommended): add a per-unit import edge list to write_manifest (a small additive change) so the entire closure is computable from manifests, and bodies load only for the final reachable set. B makes the design fully lazy, which is the whole point when there are many ln: inputs.
  • D2 — does --top XXX match the entity name or the lg= artifact name? filter_top matches get_top_module_name() (the entity), not get_graph_name() (the lg= override, lnast.hpp:468-470). So an lg-renamed top is not selectable by its lg name today. scan_pubs should index both; decide whether --top may match the lg name and keep the index and filter_top consistent. (Likely yes — the artifact name is what a user thinks of as the module.)
  • D3 — selective ln: body load (later). If ln: dirs hold many units and you need one, a load-one-unit-by-name path (load_body is already per-tree, just not exposed by name) avoids loading siblings. Treat as an optional optimization after the dir-level pruning lands, not part of v1.
  • D4 — scan_pubs lexer-only vs full streaming parse. The user accepts parsing .prp, but for the Phase-0 index (run over all .prp inputs) a lexer-only scan is much cheaper and is a trivial mirror of scan_imports. Recommended: lexer-only scan_pubs for the index; full parse only the reachable files in Phase 2.

Traps

  • The reachable set must be a superset of what is actually instantiated. Over-opening is fine; under-opening is a missing-module miscompile. When the seed-finder or the closure is unsure, include. Equivalence with today's eager path is the correctness bar.
  • Imports are over-approximated, by design. scan_imports / collect_imports return imports under if false { … } and nested scopes (collect_imports walks the whole pre_order, lhd_kernel.cpp:1955-1991). The closure may open a few files a constprop-dead branch would have skipped — acceptable, and it matches the existing behavior. Conditional imports cannot be pruned without elaboration.
  • Re-export bundles need no special handling at the index level. A pub const my_reexport = (f1=my_exp1, f2=xx) is recorded simply as the file exporting my_reexport; its dependency on the imported xx is captured because xx = import("…") is one of that file's imports. The transitive closure over imports pulls the re-exported pieces in for free — do not try to resolve bundle members in the index.
  • Names are case-sensitive / FS-independent. Import resolution deliberately does not let import("stem") bind a file named Stem.prp (find_prp, lhd_kernel.cpp:1639-1666). Keep scan_pubs and the provider index exact-match — consistent with the case-sensitive-names rule already in the frontend.
  • No freshness check on reused ln: interfaces. content_hash is written but never validated on load; restore_unit_meta trusts persisted io_meta blindly (lhd_kernel.cpp:711, 814-858). This task inherits that hazard but does not fix it; flag it if D1 grows the manifest's authority.
  • Ambiguity stays an error. If XXX (or any followed import name) is provided by two distinct inputs, keep the existing import-ambiguous behavior (lhd_kernel.cpp:1724-1747) — never silent first-hit — now detected at index time instead of after parse.

Tests

  • Equivalence (the correctness bar): for a project with several .prp files and several ln: inputs, lhd compile --top XXX … through the lazy pre-pass emits byte-identical Verilog (and identical ln:) to the eager path. Run across a top that reaches all inputs (no pruning) and a top that reaches a subset.
  • Pruning proof: a multi-ln: case where XXX reaches only some dirs — assert (via a load counter or a log line in load_ln_dir) that the unreachable forests are never body-loaded and the unreachable .prp files are never parsed.
  • Seed-not-found: --top XXX with no input declaring XXX errors fast, before any parse/load, with a precise message listing where it looked.
  • Ambiguity: XXX (or a followed import) provided by two inputs → import-ambiguous at index time, candidates listed (parity with lhd_kernel.cpp:1724-1747).
  • lg-rename selection (D2): a top declared pub comb f::[lg="chip_top"] selected by whichever name D2 settles on, resolved without parsing unrelated files.
  • Over-approximation safety: an import under if false is still followed (no under-open); a re-export bundle's pieces are reached through the file's imports.
