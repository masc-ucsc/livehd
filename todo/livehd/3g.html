3g Finish generics <T>

The generics binding machinery landed 2026-06-12 as task 2f-generics (its page is deleted; this page was stale until 2026-07-07 and used to call generics an unimplemented placeholder — it is not). What remains is finishing the contract now written in 06-functions (spec settled 2026-07-07, see "Spec decisions" below): a generic name binds any compile-time entity — a type, a comptime constant, or a lambda — and unlike an input it can constrain the types of other inputs and of outputs.

Spec decisions (2026-07-07, settled with Jose — do not re-litigate)

  • Generics subsume the [...] comptime-parameter slot. The old COMPTIME field (comb addn[n:u8=1](a:u8) / addn[3](…)) is deleted from 06-functions (it never parsed — declaration or call). A caller-picked comptime constant is a constant-valued generic; to replace the slot's defaults, generics gain declaration defaults (<T, N=4>).
  • Call-site binding follows the argument-naming rules. Named (f<T=u8, N=8>(…)) unless unambiguous (single generic / kind-unique / identifier matches the generic's name). This REPLACES the landed positional all-or-nothing rule, so fcall-generic-arity semantics change (partial lists become legal when named/defaulted/inferable).
  • Pure comptime-macro expansion. Whatever is legal for the substituted entity is legal for the generic, nothing more. Corollaries: T.[bits] is NOT a generics feature (it is exactly as (il)legal as u8.[bits], which errors today — type-name attribute queries are a separate, unspecified feature; verified 2026-07-07 that even value-side a.[bits] in a comb body hits unhandled-node for concrete types too, so nothing generic-specific to do). Constant- and lambda-valued generics are never inferred (no :T slot) — explicit or defaulted only.
  • Constraints are out of scope. No where, no <T does …> clause — the documented idiom is cassert(T does Addable) in the body, which needs nothing beyond body binding of T (subtask A) plus existing comptime asserts.
  • Input defaults join the general contract (comb f(in1:u4, in2=3)): the default evaluates in the parameter tuple's scope — 04-variables already documents the tuple-scope form b:signed=a+5 (default reads an EARLIER parameter at runtime) — and a generic name (in2=N) is just one legal default expression. Today omitting errors fcall-missing-arg.

Landed (verified against HEAD lhd, 2026-07-07)

  • Signature :T binding on params and outputs: explicit f<u8>(…), inference by unification over the actuals' declared types (literals contribute kind only), named user types (f<Byte>), comb inline re-typing, mod/pipe per-binding specialization with readable mangling (madd__u8_u8), diags fcall-generic-arity / fcall-generic-mismatch.
  • Grammar already accepts constants and lambda names in <…> (f<3>, f<inc> parse and pass arity) — but see the D validation hole below.
  • Seams: upass/runner/upass_runner.cpp — resolve_generic_binds (~:4323), the __generic_arg call-site marker (~:269), specializer clone clears generics (~:4236); lnast/lnast.hpp generics_. Front-end is prpparse (this page used to cite prp2lnast — gone).
