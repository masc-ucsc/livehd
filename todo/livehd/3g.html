3g Generics <T>

Type-parameterized lambdas (comb add<T>(a:T, b:T) -> (r:T)). Needs a specification first — this task is a placeholder so the deferred work has a home; not urgent. Successor to the generics slice that task 1p (deferred lambda templates, fully landed 2026-06-08) explicitly carved out.

Depends on

Nothing hard. The template + call-site-specialization machinery it builds on is on master (1p): a not-fully-typed lambda is a template with no LGraph; comb inlines, pipe/mod specialize per-signature into a dedup'd Sub with a readable mangled suffix. Spec-wise it interacts with the typesystem (does, declared-type propagation), so write the spec against the current 06-functions contract.

Subtasks — pending

  • A — write the spec. Syntax and semantics: where <T> binds (params, outputs, body locals, T.[bits]-style queries), inference vs explicit instantiation at the call site, constraints (T does …?), multiple parameters <T, U>, and how a generic differs from today's untyped-template behavior (an untyped param already specializes on the actual's declared type — the spec must say what <T> adds: same-type unification across params/outputs, named type reuse in the body, …). Lands in the Pyrope docs, then split implementation subtasks here.
  • B — implementation. Per-T body substitution in the specializer + the reserved mangling (add6__T_u22). Scope after A.

Subtasks — landed

None here. The seams 1p left in place (all on master):

  • prp2lnast parses the <T, U> list into the func_def generics tuple (prp2lnast.cpp:2785), with no read-site refs minted for the type names.
  • Lnast::generics_ + get/set/has_generics() (lnast/lnast.hpp:170); func_extract copies the names onto the extracted Lnast and any generic signature is stamped template_ (no LGraph) (upass_func_extract.cpp:66,491).
  • The specializer clone clears generics (upass_runner.cpp:2121) — i.e. today a generic call rides the untyped-template path with no per-T substitution or unification; the __T_ mangling is reserved, unimplemented.

Design notes

None yet — intentionally unspecified. When picking this up, start from the 1p decisions that already constrain it: specialization keys on the actual's declared type (an untyped actual is a compile error, fcall-untyped-actual); specialized modules dedup by mangled name; readable suffix mangling. The open design question is what <T> buys over the implicit untyped-param specialization that already works.
