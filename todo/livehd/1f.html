1f Source locator migration (SourceId spans, located diags)

Replace today's per-node source attributes — a 21-byte lnast.loc struct plus an uninterned per-node std::string filename — with one uint64 SourceId per def-bearing node/dpin, resolved through the HHDS Source_locator table (hhds-srcloc). Carry it through every copy/clone/lowering path (killing the hand-maintained 8-node-kind loc whitelist), mint combined IDs where one value derives from several source sites, convert the ~70 null-span diag sites, and delete the old attributes.

Depends on

  • hhds-srcloc — the Source_locator class, its per-graph/per-tree ownership, and its Forest/GraphLibrary save/load/merge integration land in ../hhds first. 1f is the LiveHD-side migration onto it.

Inbound: Pyrope 2p (source-derived SSA names reuse the SourceId), LiveHD 3f-B (full-fidelity source_id spans), 2n (LSP span precision), 5c (stable signal names), Verilog 5e (cgen // src: comments) all consume what 1f delivers.

Decisions (2026-06-09 review)

  • One integer, not (file, loc) payloads. Nodes/dpins carry a single uint64 SourceId; the locator table owns the file/span/line payload. Today's cost is ~24 B + a full filename string copy per located node.
  • Hash-minted IDs. SourceId = 64-bit FNV-1a of (canonical path, start_byte, end_byte), so the same span gets the same ID in every artifact and merges dedup for free; only true hash collisions probe-remap (locator-local). Avoids the frequent-remap problem of sequential IDs when artifacts combine.
  • Ownership matches the single-writer unit. One locator per lnast/graph (LiveHD multithreads across artifacts, never within one — no locks needed). The per-Forest/GraphLibrary locator is materialized only at save/merge as a union; loads expose it as a shared read-only base.
  • The class lives in hhds, serialized natively by Forest::save/GraphLibrary::save/load_merge (so derived trees' provenance survives ln:/lg: round-trips without manifest plumbing).
  • Combined IDs instead of ID sets. When one value maps to several source sites (inline = callee def + call site, merged writes, aliases), combine(a,b) mints a new deterministic ID with parents; consumers keep tracking one integer. Diags render the primary anchor as the span and the rest as Diagnostic::notes — the notes channel is fully plumbed in both renderers today and has zero producers.
  • Lean internal model; ECMA-426 as egress only. The locator stores anchors (+ combine parents) — nothing else. ECMA-426 .map files are a lossy output projection for external visualization/tooling compatibility, not the canonical store. SymbolId / ScopeId / quality-marker / reverse-query machinery from the older draft is dropped from scope (see the end of this page).
  • 1f owns the diag sweep. After migration, the ~70 null-span error sites get located, and lnast.loc / lnast.fname / livehd::attrs::loc / livehd::attrs::source are deleted.

Design notes

Resolution flow

A pass holding a node/dpin reads its uint64 SourceId attr and asks the artifact's locator: resolve(id) → primary anchor {path, start_byte, end_byte, line}; resolve_all(id, depth) → all anchors (combine parents expanded, depth-capped). Line:col and end_line:end_col derive from the per-file line-offset table. Diag emission fills Span.source_id + the resolved fields; extra anchors become notes. Nothing in the IR or the compile depends on the locator — losing it degrades diags, never semantics.

What gets an ID

Definition-level granularity: nodes that define a destination (assign/store/declare, tuple_get, comparison/logic/arith ops, func_call, …) plus the statement anchors already covered today (if/while/cassert/range/attr_set). Operand refs carry nothing — provenance of an operand is the provenance of its defining node. On LGraph, def-bearing artifacts are cells, registers, memories, ports, and Sub instances; node-level is the default, pin-level only where finer attribution is actually needed.

SSA and derived names

All SSA versions of a are distinct defs with distinct SourceIds (each points at the assignment that produced it). Pyrope 2p derives readable SSA/tmp names (foo_l42_a) from the same IDs, and 5c builds its optimization-stable signal names on top.

Egress: ECMA-426 (output compatibility only)

For textual egress artifacts (top.v, generated Pyrope), emit a standard version-3 source map (top.v.map + //# sourceMappingURL=) so stock JS/TS-ecosystem tooling and code visualizers can jump generated→original. The projection is lossy on purpose: ECMA-426 maps one generated position to one original position, so combined IDs export their primary parent; LiveHD-only data (full parent lists) stays in the locator table, referenced under x_livehd. The locator's own serialized form (hhds-srcloc) is the canonical store; there is no separate canonical JSON sidecar.

Dropped from scope (future extensions, only if a consumer demands them)

The earlier draft of this page (salvaged from the retired docs/contracts/sourcemap.md) specified SymbolId + ScopeId tables, seven derivation kinds, attribution-quality markers (exact/alias/fallback/…), tombstones for folded/removed defs, reverse queries (source span → IR nodes), and a canonical .lhsm.json sidecar. All dropped: the lean anchor + combine model covers diags, LSP, naming (2p/5c), and egress comments (5e). If a future consumer needs symbol identity or fold tombstones, they become new locator entry kinds — the append-only hash-ID scheme leaves room.
