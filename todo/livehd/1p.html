Design notes

Revised understanding (2026-06-08)

The contract's deferred-limitation #1 — “for x in args does not lower because runtime-tuple-iteration is unsupported” — is incorrect. Verified: the runner's unroll_for var-arg branch (upass_runner.cpp:2742) unrolls for a in args correctly during a comb inline; a comb add_all(...args){ mut acc=0; for a in args { acc=acc+a }; r=acc } lowers to z = a+b+c with no runtime tuple. The real blocker is a walk-order corruption of the template body: the main upass walk (pass_upass.cpp:401) lowers every unit standalone, including the var-arg template. Standalone, args is unbound, so the for is emitted verbatim and the statement after it (r = acc) is dropped; that corrupted body is swapped in-place (:438). Because set_function_registry is rebuilt per unit (:422), a later caller sees an add_all with no output write → not inlinable → the call survives to tolg and errors. Defining the caller before the template dodges it (which proves the diagnosis). The fix belongs with the 2u for-unroll work (branch task-2u-loop-migration): skip lowering a template body in the main runner walk — as tolg/timecheck already do (pass_upass.cpp:473) — so the body stays pristine for the call-site inline/specialize. 1p-F below assumes that 2u-for fix is in.

Tests (staged, failing until F lands)

Three repro/spec pairs are in place under inou/prp/tests/equiv/ (auto-registered prp-equiv-* targets), all currently failing on the errors above:

  • mod_varargs_csa.{prp,v} — the convoluted one: a var-arg mod csa(...args) whose for a in args calls a second not-fully-typed mod blk_add(b) that also loops (a block adder — sums b's bsz-bit blocks via for i in 0..<(b.[bits]/bsz)). Both levels must unroll; LECs to z = a+b+c. Fails first at the outer var-arg boundary (1p-F); the inner mod surfaces three more "runner/specialization should handle" gaps once F lets it through: (1) b.[bits] does not fold inside a mod body — the attr_get survives to tolg (the comb inliner folds it via the declared type, untyped_param_loops_runner_unroll; a specialized mod clone must do the same); (2) an untyped mod output -> (s@[0]) specializes but derives to 1 bit instead of the body's width (so the inner adder needs an explicit output type today); (3) shift/mask arithmetic (>>/&/ <<) inside an unrolled mod loop leaves unresolved temps / a stray tuple_add.
