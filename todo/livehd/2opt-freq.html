2opt-freq Agent-in-the-loop frequency optimization

A continuous optimization loop with a coding agent in the middle: the agent edits a Pyrope or Verilog design (and, for Pyrope, inserts block-scoped attributes that steer ABC), then lhd compile → lhd lec gate → lhd pass color/abc → QoR score. Only LEC-passing edits survive; the objective is post-synthesis clock frequency. The external agent drives the stateless lhd CLI directly (JSON result envelopes) — 2opt-freq delivers kernel primitives, not a driver, and does not depend on the 3y-agent @tag layer. The mechanical backbone already exists and is test-proven (lhd/tests/lhd_abc_test.sh is the iteration body, including the negative control); what is missing is the objective function (no timing read-back anywhere), the source-scoped steering channel (no attribute reaches pass.color/pass.abc), and per-region control.

Contract rulings (settled 2026-07-12)

  • Gate is strictly cycle-accurate in v1. Per agent edit: lhd compile must pass; optionally lhd sim when tests exist; then lhd lec of the edited source vs the previous accepted source (chained RTL-to-RTL equivalence — transitive, and the formal_cache.json verdict cache + semdiff structural skip make small diffs cheap). Retiming and latency changes are off-limits; in the future, pipelining becomes legal only inside fluid once 4f-fluid-lec transaction-level equivalence exists.
  • Netlist-vs-source LEC is an lhd-bug catcher, not the loop gate. Mapped-netlist LEC (per-region vs its pass partition twin, with pass liberty gensim cell models) runs in tests only — it validates pass.abc, not the agent's edit, and is a waste per-iteration when there are no lhd bugs.
  • Tiered timing oracle. Phase 1: ABC's own post-map delay, extracted per region in-loop (subtask A). Phase 2: OpenTimer STA as the accurate whole-design scorer (subtask D — moved here from 3y-agent item G).
  • Attributes bind by lexical block. Grammar extension of the form {::[abc="…", color="…"] a = b + c }: the annotated block becomes its own synthesis partition group — pass.color coded in Pyrope — and may carry per-region ABC directives. Attributes are Pyrope-only; Verilog designs get code edits plus global --set pass.abc.* knobs.
  • Benchmarks (never committed): ../simplechisel/build_dual (generated Pyrope + Verilog) and xs_core_prp / ../xs/repo/build/rtl (Verilog).

Bugs found and FIXED by the loop demo (2026-07-12)

  • pass.abc const-collapse miscompile (fixed). y = ((a + b) * 342) >> 10 mapped every output bit to a _const0_ tie cell (y≡0), any library, no diagnostic. Root cause: the Mult/SRA bit-blasts' effective-width path sized CONSTANT operands via bits_of(pin) — const pins usually carry no bits attr, so real_width floored to 1 and the const 342 was read as its bit 0 (=0), zeroing the product. Fix: const pins bypass the eff-width clamp (exact two's-complement bits via abc_bit) and eff_width sizes consts from their value. Regression: abc_constmul.prp fixture + a constmul section in lhd_abc_arith_test whose oracle is the netlist-vs-twin LEC (validated negatively: reverting the fix REFUTES with a concrete counterexample). Exactly the bug class the tests-only netlist LEC exists for — the loop gate (source-vs-source) was never at risk.
  • Blind partial scores (fixed). A region with a blackboxed div/mod under-reports gates/area/delay; qor.json now carries div_blackbox per region + total and the summary line says [PARTIAL: N blackboxed div/mod cone(s) unscored], so an agent never trusts a blind score (the demo's div→reciprocal-multiply iteration is the canonical response).
