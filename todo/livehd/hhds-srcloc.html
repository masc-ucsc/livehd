HHDS Source_locator — hash-keyed source-anchor table

A compact provenance table mapping a 64-bit integer SourceId to a source anchor (file + byte span + line). Lives in ../hhds (source_locator.hpp) next to Forest / GraphLibrary and serializes with them. IR nodes then carry one uint64 attribute instead of a full filename string + 21-byte loc struct per node. Design notes

ID minting — hash, not sequence

SourceId = 64-bit FNV-1a hash over (canonical file path, start_byte, end_byte). The same source span therefore yields the same ID in every locator, in every run, in every separately-compiled artifact — so when two locators are combined the union dedups for free, and bazel-parallel compilations agree on IDs without coordination. This is the same move as the GraphLibrary gid refactor (FNV-1a(name), pick_gid_for_name linear probe in hhds/graph.hpp).

Canonical path = workspace/exec-root-relative, never absolute — matching the depfile path contract (lhd/lhd_kernel.cpp:706-709). Absolutizing (what Hhds_graph_library::canonical_path does today via fs::absolute) would make IDs differ across checkouts/exec-roots and break exactly the cross-artifact agreement claimed above.

Collisions: if a mint finds the hashed slot occupied by a different payload, probe-remap to a fresh nearby ID. Remapped IDs are locator-local (they lose cross-artifact determinism), which is acceptable because at 64 bits genuine collisions are essentially theoretical; the remap path exists for correctness, not as a hot path. 32 bits would not be enough — the birthday bound (~77k entries for a 50% collision) is below the def-count of a real design.

Entry model

  • Span anchor (the common case): {file_id, start_byte, end_byte, line}. line is stored so resolution never requires re-reading the source file.
  • Line-only anchor: {file_id, line} — for ingress that only knows file:line (e.g. yosys Verilog attributes).
  • Combine entry: {parents: [SourceId…]} — see below.
  • File table: file_id → {canonical path, optional content hash, optional line-offset table}. The line-offset table (byte offset of each line start) lets a resolver derive line:col and end_line:end_col from byte spans — full LSP-grade intervals — without storing columns per entry.

Combined IDs (one value from N source sites)

Transformations routinely map one IR value to several source sites (alias, inline = callee def + call site, merged writes). Rather than tracking sets of integers at every consumer, combine(parents…) mints a new ID meaning "all of these anchors": entry kind combine with the parent list. The parent list is caller-ordered and order-significant — the first parent is the primary anchor (what a diagnostic renders as its span; the rest become secondary notes), and the ID is the hash over the parent IDs in that order, so combine(a,b) ≠ combine(b,a) but re-minting the same ordered list anywhere dedups to one entry. The call is variadic: a consumer accumulating N sites (e.g. N contributing writes) collects them and mints once — never builds association-sensitive combine(combine(a,b),c) chains. resolve(id) returns the primary anchor; resolve_all(id, max_depth) expands parents with a visited set (linear in reachable entries — combine graphs are DAGs and diamonds must not re-expand) plus a depth cap, reporting truncation rather than silently dropping anchors.

Ownership and threading

LiveHD multithreads across lnasts/graphs, never within one — only one thread edits a given graph/tree at a time. The locator must match that single-writer unit: one instance per artifact wrapper — a member on Graph, and per-Lnast on the tree side — so mints need no locks. A per-GraphLibrary/Forest locator exists only as the serialized union built at save time. After a load, the library-level table is a shared read-only base; each artifact layers its own initially-empty delta locator over it (lookups consult own-then-base).

Why the wrapper and not the raw tree/forest: the LiveHD tree lifecycle defeats both naive homes. The runner's materialize cycle swaps each Lnast body for an unattached temp tree (create_tree_temp → replace_body), so a per-Tree member would be silently dropped every rebuild; and ln: load adopts all units from one shared forest (Lnast::adopt), so a mutable per-Forest member would be multi-writer. A locator hanging off the Lnast (holding an hhds::Source_locator) survives body swaps for free; the seams that must hand it over explicitly are export_into (clone into the save forest) and the save-time union below.

Serialization and merge

  • One table file per save dir, next to library.txt / forest.txt (binary or text — implementer's choice, but append-friendly and versioned).
  • save = union of the per-artifact locators, where each operand is the artifact's own-then-base view — base entries reachable from a loaded artifact (including combine parents) must be carried into the union, or an ln: load → no-op → save round-trip silently drops all provenance (the kernel's save_ln_dir exports into a fresh forest).
  • Union algorithm: dedup by payload first (anchor equality — two locators that probe-remapped the same colliding spans in opposite orders must still converge to one entry per span), then probe-remap residual true ID conflicts. The remap is applied to (a) the registered uint64 SourceId node/pin attribute — hhds owns the attr tag so save/load_merge can rewrite the typed attr maps, exactly like the Sub-ref gid remap — and (b) combine entries' parent lists: a rewritten parent list re-keys its combine entry (ID = hash(parents)), which can cascade; process in topological parent order (append-only DAG ⇒ terminates) and fold the re-mints into the returned remap table.
  • Saving into an existing dir: the on-disk table is an immutable base — Forest::save skips clean (non-dirty) trees, whose serialized attrs can never be rewritten, so only fresh entries may be remapped; IDs already used by a previously-saved tree never move.
  • load_merge (the lg: linker path) unions the two libraries' tables the same way.
  • Entries are append-only; existing entries never change meaning.

API sketch

class Source_locator {
  SourceId mint(path, start_byte, end_byte, line);
  SourceId mint_line(path, line);
  SourceId combine(span<const SourceId> parents); // ordered; [0] = primary;
                                                  // ID = hash(ordered parents)
  Anchor          resolve(SourceId) const;          // primary anchor
  vector<Anchor>  resolve_all(SourceId, depth) const; // DAG expand, visited set
  RemapTable merge(const Source_locator& other);   // payload dedup, then remap;
                                                   // cascades combine re-keys
  // + save/load hooks used by Forest / GraphLibrary
};
