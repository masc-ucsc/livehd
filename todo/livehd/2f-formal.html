2f-formal Formal property engine — assert/assume proving + a reusable property library (on pass/lec)

A single-graph formal-property tier built on the landed lec Encoder. Where lec is relational (graph-vs-graph miter), 2f-formal answers single-graph, sub-cone questions about one design: "is this cond always true/false?", "are pins X and Y always equal — and which bits?", "is X constant?". It has two faces:

  1. A reusable C++ property library any pass calls. First consumers: the assert/assume driver in this task, and (future) pass/memory port-sharing / banking and pass.abc optimizations that need bit-level guarantees.
  2. An always-on verification tier that proves both user assert/assume and built-in structural obligations (the first being Hotmux selector one-hotness) at compile time: proven ⇒ removed (marked, runtime check elided), disproven over free inputs ⇒ compile error, everything else ⇒ kept for a (future) runtime check, with a deferral warning.

The cascade. This slots between the two existing assertion tiers: comptime (upass.verifier constprop) → formal SMT (this) → runtime (future). The verifier already drops comptime-true asserts and errors comptime-false ones; only the unknown survivors are materialized as lgraph sub-nodes and handed here, which naturally bounds how much work the always-on tier does.

Depends on

lec — the landed Encoder only (pass/lec/encode.cpp M1/M2/M4: LGraph→cvc5 bit-vector translation, Val{term,width,is_signed}, real_width sign/width handling, the inductive flop-cut state model, memory-as-array). It does not depend on the pending lec roadmap (M3 def-diff, M5 assume-guarantee, M6 cache). Reuses the same @cvc5 dep and the QF_ABV setup. 3f-temporal (Pyrope, Group 3) depends on this for the substrate used to prove temporal assertions (assert(rose(req) implies eventually[1..=10](rose(ack)))); that page owns the temporal-operator lowering, this page owns the proof substrate. The initial single-frame engine here does not by itself prove bounded temporal behavior; that is gated on the deferred reachable-state BMC/k-induction extension.

Key realization — demand-encode once, collapse, query sub-cones

These checks are not "graph X vs graph Y". They are "sub-cone X of graph Y vs sub-cone X′ of the same graph Y", and in the common case both cones share the same flops / memory / inputs. So the engine starts at the property pins and traverses only the required backward/forward cones. Encoding is demand-driven and memoized, with full structural sharing: every visited lgraph pin maps to exactly one cvc5 term, reused by all later queries. Unrelated logic is never walked, so an unsupported node outside every property cone cannot poison the module's results. A property is then a query over the lazily built terms:

  • assert(cond) → demand the term for the cond driver pin, add the normal-assert cone-reset guard, assert ¬cond, checkSat.
  • rdaddr_i == rdaddr_j → both address pins are terms over the same upstream state, so asserting addr_i != addr_j collapses all shared flops/memory/inputs to one term each automatically. UNSAT ⇒ always equal ⇒ the two reads can share a port.
  • "which bits of X equal Y" → BITVECTOR_EXTRACT bit k from the pre-built X/Y terms, assert X[k] != Y[k], incremental checkSat per bit → a per-bit mask.

The collapse is therefore intrinsic, not an optimization: it falls out of one memo table shared across every query. A cone traversal also brings in the semantic side constraints required by its state cuts and memory reads, but does not expand into unrelated graph regions. Each query is push → assert ¬property (+ assumes) → checkSat → pop on a single long-lived solver, so overlapping cones amortize their encoding.

C++ API — single-graph property prover (mirrors lec::prove_equal shape)

New pass/formal/prove.{hpp,cpp} (target //pass/formal:formal), depending on //pass/lec:lec_query for the cvc5 Encoder and its width / sign / flop / memory machinery. The shared Encoder stays in pass/lec for now; pass/formal adds the single-design Prover. Refactor the current eager, encode-local pin2val table into a reusable lazy encoding context. val_of() demand-encodes an arbitrary internal driver pin and the required portion of its cone; it is not merely a getter over a prior whole-graph encode.

namespace livehd::lec {
// --- demand-driven extension/refactor of the existing Encoder (encode.hpp); stays in pass/lec ---
// Encodes driver_pin and the required cone on first use; later calls reuse the Val.
std::optional<Val> Encoder::val_of(hhds::Pin_class driver_pin);
}  // namespace livehd::lec

namespace livehd::formal {   // the new single-design pass (pass/formal)

// --- the new single-graph property prover ---
struct Prove_options {
  std::string engine     = "ind";   // reuse lec: flop-cut single frame (free state)
  int         budget_k   = 256;     // rlimit-per = budget_k * cone_node_count  (deterministic)
  int         cone_max   = 50000;   // pre-solve gate: cones bigger than this skip -> Unknown
};

enum class Reset_policy {
  Include,             // core-library default; property must hold during reset too
  ExcludeConeActive,   // normal assert: constrain every cone flop's reset inactive
};

using Verdict = lec::Verdict;   // { Proven, Refuted, Unknown } from query.hpp

class Prover {
public:
  Prover(hhds::Graph* g, const Prove_options& = {}, const sub_lib* = nullptr); // initializes lazy encoder

  // scalar properties (1-bit cond, or whole-signal pins)
  Verdict               is_true (hhds::Pin_class cond, Reset_policy = Reset_policy::Include);
                                                    // UNSAT(¬cond)?Proven : SAT?Refuted : Unknown
  Verdict               is_false(hhds::Pin_class cond);
  Verdict               equal   (hhds::Pin_class a, hhds::Pin_class b);
  std::optional<Lconst> is_constant(hhds::Pin_class x);   // proven-constant value, else nullopt

  // structural predicates (built-in obligations, e.g. Hotmux selectors)
  Verdict               is_onehot0(hhds::Pin_class sel);  // at most one bit set (one-hot OR zero = default arm)
  Verdict               is_onehot (hhds::Pin_class sel);  // exactly one bit set

  // bit-granularity (the ports / banking requirement)
  Bitmask bits_equal   (hhds::Pin_class a, hhds::Pin_class b); // per-bit: positions provably equal
  Bitmask bits_constant(hhds::Pin_class x);                    // per-bit: provably-constant positions (+values)

  // checked contracts + later hypotheses (assume)
  Verdict check_assume(hhds::Pin_class cond); // no reset exclusion; never assumes itself
  void assume(hhds::Pin_class cond);     // adds solver.assertFormula(cond) to every later query
  bool assumptions_consistent();         // one checkSat of the conjunction of assumes

private:
  // owns: cvc5::TermManager + Solver (QF_ABV), one lazy per-pin memo map, cone-size cache.
  // each query: push / assert ¬property + assumes / set rlimit-per / checkSat / pop.
};

}  // namespace livehd::formal

Whether the property cone touches state (a stateful query may have an unreachable witness) is reported alongside Refuted so callers apply their own soundness policy (below). The library never decides removal/errors — it returns verdicts; the driver decides.

Built-in property checks (structural obligations)

The always-on tier proves more than user assertions: many IR constructs carry an implicit obligation the formal engine should discharge with the exact same yes / no / defer-to-runtime cascade as assert. The first — and one of the most valuable — is Hotmux selector one-hotness.

  • Hotmux / index-selection one-hot. match and unique if lower to Hotmux, which assumes its selector is one-hot (an arm is chosen by sel == 2^k; if two bits are set no arm matches and it silently falls to the default/last arm — a real mutual-exclusivity bug). This check is the dual of that implicit assume: for every Hotmux driver, prove is_onehot0(sel) (at most one bit, since an all-zero selector legitimately takes the default arm). Proven ⇒ drop the obligation (the Hotmux is guaranteed safe — a downstream pass may exploit it). SAT over a stateless cone ⇒ two arms can fire at once ⇒ compile error (overlapping match/unique if conditions). Otherwise ⇒ keep a runtime one-hot check + emit the deferral warning. The dynamic tuple-index Hotmux (decoded idx==k match-chain) is one-hot by construction and should prove instantly; user unique if is where this earns its keep.
  • No frontend work — ships first. Hotmux nodes already exist in the lgraph, so this is the lowest-friction first always-on application: it needs only the driver skeleton, the proven/runtime_check attrs and the deferral warning — none of the assert grammar / tolg materialization. It lands before the user-assert driver (see milestones).
  • Same framework for future obligations. The descending bit-range precondition (hi ≥ lo, today lower_range_assert/ lgassert) becomes a discharged obligation; likewise div-by-zero, cast-overflow, array-index-in-bounds — each is just another built-in predicate run through the same prove / error / defer cascade.

Design decisions (locked)

  • Demand-encode once + collapse. Start from each point of interest, traverse only the required cone, and memoize each visited pin/state term once. Overlapping queries share terms via val_of() and one incremental push/pop solver. Do not eagerly call whole-module Encoder::encode(g): that leaves encoding cost unbounded by cone_max and lets unrelated unsupported nodes poison every query.
  • State = flop-cut, single frame. Reuse lec's engine=ind: flops/memory are free symbolic state, collapsed to one shared term each. A UNSAT over free state is sound for all states, including reachable.
  • Asymmetric soundness policy (assert).
    VerdictCone has flop/memory?Driver action
    UNSAT (Proven)eithermark proven; cgen elides the runtime check
    SAT (Refuted)no (free inputs ⇒ reachable witness)hard compile error + counterexample + the assert message
    SAT (Refuted)yes (witness may be unreachable)keep + mark runtime_check (no error)
    Unknown / budget-exceededeitherkeep + mark runtime_check
    The invariant that makes always-on safe: hard-error only on a definitive SAT over a stateless cone (after applying assumes and, for a normal assert, its cone-reset guard). Budget / Unknown / stateful-SAT never error — so a budget mistune can never turn a good build into a false error; worst case it under-proves and leaves a runtime check.
  • Normal assert excludes active reset, and only assert does so. While collecting a normal assertion's cone, gather each encountered Flop's reset_pin and negreset, build its reset-active expression, and constrain it inactive for that query. Modules containing registers must expose explicit reset pins. Memory has initialization, not an intrinsic reset, so it contributes no reset guard; logic may still use a module reset to initialize an SRAM through normal writes. General library queries, built-in obligations, and checked assumes include reset-active valuations. assert_always also uses Reset_policy::Include — it must hold every cycle, reset included (that is the only difference from a normal assert).
  • Deterministic budget = cvc5 rlimit-per, cone-scaled. Per query, rlimit-per = budget_k × (#nodes in the effective query cone), including assumption and reset-guard cones used by that query. cvc5's rlimit is a deterministic internal resource counter (rewrites/decisions/conflicts), machine- and wall-clock-independent — the realization of "#statements passed to formal × constant = max clauses to eval". A cheap pre-solve gate skips cones above cone_max straight to Unknown. Same config ⇒ identical verdicts everywhere.
  • assume = checked contract + optimization hypothesis. Every in-scope assume(cond) is assertFormula'd before assert and LEC queries and is also fed to ABC as an optimization constraint. It is itself a formal obligation: check it without assuming itself or relying circularly on unproven peer assumptions. Proven ⇒ elide its runtime check; definitive stateless refutation ⇒ compile error; stateful refutation or Unknown ⇒ keep a runtime contract check and emit formal-assume-deferred. Optimization may still exploit the declared contract: behavior outside it is undefined, and the runtime check detects a contract violation but cannot restore logic optimized away under the assumption. The engine also runs one assumptions_consistent() check per scope — a jointly-contradictory assume set is a warning (it would make every proof vacuous) and suppresses removals.
  • The library returns verdicts; drivers own policy. The assert driver applies the table above. A future pass/memory asks equal(addr_i,addr_j) and applies its own conservative fallback (can't prove ⇒ use a separate port; error only when the un-mergeable count exceeds the declared rdports limit). No reachability concern there — the fallback is "more ports", never a wrong answer.
  • Deferred-to-runtime ⇒ warning (silenceable). Whenever any obligation — user assert/assume or a built-in like Hotmux one-hotness — is deferred to runtime (Unknown / budget-exceeded / kept stateful-SAT), emit a diagnostic warning: it now costs a runtime check formal could not discharge. Per-category codes (formal-assert-deferred, formal-onehot-deferred, …) so each is independently silenceable — a global --set pass.formal.warn_deferred=false and per-category --set pass.formal.warn_<cat>=false turn them off. Proven obligations are silent; errors stay errors. Reuses the existing diag warnings/ harness (:type: warning).

Pipeline placement

  • pass.formal is its own pass — single-design reasoning, distinct from pass.lec (two-design equivalence). It owns the Prover library + the always-on drivers and orchestrates other passes (feeds proven contracts to pass.abc, will call a future pass/memory). It reuses pass/lec's cvc5 Encoder via //pass/lec:lec_query; that shared encoder may later relocate to a neutral target consumed by both.
  • Pipeline slot. Runs after bitwidths are final and before codegen: lnast → lgraph (tolg) → cprop → bitwidth → pass.formal → cgen. The cvc5 encoding needs resolved widths/signs, so pass.formal must follow bitwidth; its proven contracts must be available to a later pass.abc, and its removals/marks must land before cgen.
  • Always-on, opt-out. pass.formal is in the default compile recipe, opt-out via --set compile.formal.enabled=false (consistent with the bare-compile-max-diagnostics philosophy). The deterministic cone-scaled budget keeps build time bounded, and Unknown/budget never errors, so always-on is safe.
  • Library: caller-agnostic Prover in pass/formal. Always available to any pass.
  • Future callers (pass/memory ports/banking, pass.abc) invoke the library directly to decide ports / memory / banking once they need bit-level property guarantees — their own pages.

ABC integration — proven assumes as don't-cares

A formally established assume is an input/state constraint: states that violate it never occur, so they are don't-cares pass.abc can exploit to synthesize smaller logic. For each discharged assume, pass.formal records whether it was proven (UNSAT of its negation) or merely declared (kept as a runtime contract), and hands the constraint set to pass.abc.

  • --set pass.abc.use_proven_assume=true (default true). Feed only formally proven assumes to ABC as care-set constraints. Sound by construction: the assumption provably holds on the design, so optimizing under it changes nothing observable.
  • --set pass.abc.use_all_assume=false (default false). When enabled, also feed declared-but-unproven assumes (those deferred to runtime). Aggressive mode: behavior outside the contract becomes undefined and the runtime check can detect a violation but cannot recover logic optimized away. Off by default — it trades soundness-without-the-contract for area.
  • Mechanism. Each fed assume constrains its bit-blasted cone true in ABC's network, so violating minterms become don't-cares for the AIG / tech-map flow. pass.formal emits the constraint list; pass.abc consumes it under the flags above (both off ⇒ today's exact behavior).

Traps

  • Width = magnitude+1. Reuse real_width / fit_to from encode.cpp for every cond/pin — do not re-derive widths (the bit-width trap that silently breaks every cvc5 comparison).
  • Apply assumes before erroring. A stateless cone can still be input-constrained by an assume; a SAT may vanish once assumes are added. Never hard-error before the in-scope assumes are assertFormula'd.
  • Do not prove an assume with itself. An assume is both an obligation and a later hypothesis. Its obligation query must not contain that same assumption or depend circularly on unproven assumptions; otherwise every assume proves vacuously.
  • Reset masking belongs to normal asserts only. Collect reset polarity from cone Flop nodes and constrain those resets inactive. Do not infer a reset from a Memory's init, and do not silently add reset guards to Hotmux, memory/ABC, LEC, or assume queries.
  • Consistency before vacuity. Contradictory assumes make every assert vacuously Proven — run assumptions_consistent() first and suppress removals (warn) if UNSAT.
  • SAT ≠ bug when state is in the cone. Only stateless-cone SAT is a reachable counterexample; stateful SAT must keep, never error.
  • Budget never errors. Unknown/budget-exceeded ⇒ keep+runtime, so the set of compile errors stays deterministic regardless of budget_k.
  • One-hot = onehot0 (≤1), not ==1. An all-zero Hotmux selector legitimately takes the default arm, so the obligation is "at most one bit set" — build it as (sel & (sel-1)) == 0 (covers zero and exact one-hot in one term). Requiring exactly-one would false-error every design that relies on the default arm.
  • Only survivors reach here. The verifier already discharged comptime-constant asserts; don't re-prove them — tolg only materializes the unknowns.

Deviations from the plan above (intentional, documented)

  • Kind carrier = sentinel const child, not the node name. The cassert NODE name does not survive upass re-emission (its CHILDREN do), so the kind rides a '__fkind__assume' / '__fkind__assert_always' const child ahead of the user message.
  • User asserts defer on refutation (no hard error yet). Only the Hotmux structural check hard-errors. A normal assert may legitimately fail during reset, so the Reset_policy::ExcludeConeActive reset-exclusion is a prerequisite for erroring user asserts — deferred. Refuted/Unknown user asserts keep a runtime check.
  • Only PROVEN assumes are assert hypotheses. Each assume is proved independently (empty hypothesis set ⇒ no circular self-proof); only proven ones feed the assert queries (sound), avoiding unsound assert removal under an unproven contract.
  • Encoder kept self-contained, not refactored. The demand encoder lives in pass/formal reusing lec's exported free functions; the planned in-place refactor of encode.cpp into a shared lazy context is a future de-dup.
  • ABC don't-care exploitation is the follow-on. The flags + gathering ship; building the EXDC don't-care network that hands the constraints to ABC's mfs/fraig is the remaining ABC-integration step.
