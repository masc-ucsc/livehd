Scope decisions (2026-06-09):
    • In scope — core: single-dimension reg mem:[N]T = init async memories with runtime-index reads/writes → one Memory cell (Pyrope default: 0-cycle read, forwarding on; spec 08-memories).
    • In scope — __memory(cfg) RTL instantiation: the direct pin-level cell call; doubles as the shared lowering seam below. cfg vocabulary = the cell pins verbatim (decision 2026-06-09): addr/bits/clock_pin/din/enable/fwd/posclk/type/wensize/size/rdport + the new init — no latency field, type picks 0 async / 1 sync / 2 array, rdport entries are strictly 0/1, read results come back as a tuple indexed by read-port order (res[0], res[1]). The ../docs pyrope/08-memories.md RTL-instantiation example must be fixed with the implementation (latency→type, clock→clock_pin, rdport (-1,1,0)→(1,1,0)); latency-style sugar can live in the stdlib later.
    • Common codegen interface: runtime-indexed mut arrays are NOT memories (no cross-cycle persistence) but lower through the same __memory interface configured with zero cycles / no clock (type=2 array) — one code path emits the Memory cell for both. const ROMs too (decision 2026-06-09): a runtime-indexed read of a const array lowers through the same interface — type=2, init = contents, read ports only, zero write ports; a later LG pass may turn small ROMs into mux trees if profitable.
    • When the Memory path triggers: an array-typed reg (reg mem:[N]T) always lowers to a Memory cell, even if every index is comptime (verified 2026-06-09: comptime-indexed reg arrays are broken today — tolg "tuple/field store not handled", so this is greenfield, no Flop-path regression; memory↔flops conversion is the future LG pass's job). mut/const arrays take the Memory path only when some index is runtime — fully-comptime accesses keep today's tuple resolution. Whole-entry writes only (wensize=1) on the inferred paths; a partial-entry write (mem[i]#[r]=…) is a compile error this slice (__memory passes wensize through verbatim).
    • Reset / initialization (decision 2026-06-09): a clocked reg memory accepts only = nil / = 0sb? in this slice (no reset hardware; random pre-reset contents per spec) — a concrete reset value (reg mem:[8]u8 = 3 or a tuple literal) is a clean compile error for now (the reset-sweep FSM is a later slice). A runtime-indexed mut array must accept concrete init (mut mem:[4]u8 = (1,2,3,4) — its init is the per-cycle default, no reset FSM involved). Mechanism: add a new comptime init sink to the Memory cell (graph/cell.cpp) — easy to drive from prp/upass; cgen Verilog emits it readmemh-style (initial contents / per-cycle default for type=2). Watch-out: a 12th sink changes the per-port pid stride 11→12 (pid%stride wrapping; "11addr"-style names) — audit inou/yosys, cgen, and the 05-lgraph doc table when adding it. Encoding: one wide const, entry 0 in the low bits, row-major.
    • Memory types + timecheck model (decision 2026-06-09): __memory supports both async (type=0) and sync (type=1) — the graph Memory cell already handles both; verify type=2 (array) end-to-end in graph/cgen and add it if missing. From a timecheck point of view a memory is treated like a register: async = read returns the committed state immediately (a reg read), sync = the extra cycle is "time the write takes to commit". Latency is pinned at 1 (the only latency the cell supports; Chisel-style read latency 2,3,… is out of scope).
    • Port policy (decision 2026-06-09): one read port per read site and one write port per write site (post-unroll), no muxing/exclusivity analysis in tolg — write enables come from control flow as-is. Same-cycle conflict behavior is defined by the memory config (fwd or not), not by tolg. Port reduction/merging (spotting exclusive or conflict-free ports and shrinking the memory) is a future LGraph pass — explicitly not this slice.
    • Out of scope: regref shared memories, multi-dimensional memories, sync-memory inference (flop-the-inputs/outputs recognition) — async + explicit flops covers those tests for now.
