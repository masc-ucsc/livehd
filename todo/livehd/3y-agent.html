Design notes — the @tag layer (item B)

Everything below is sugar over the stateless kernel: a @tag is a named, persistent bundle of (inputs, config, state, run history), and lhd run <command> @tag resolves the bundle and invokes the same kernel commands (compile/sim/lec/pass, recipes O0|O1|O2), then appends the kernel's result envelope to the tag's run log. The tag layer adds no semantics, only persistence — an agent can always drop to the kernel for a one-shot.

@tag sigil

Tags are always prefixed with @, so the CLI parser and the calling agent never confuse a tag with a command or option. A tag is a plain name or a path:

  • @tst1 → stored under <state>/tags/tst1/ (the state root belongs to the tag layer alone; the kernel stays cache-free)
  • @./my_build → stored at ./my_build/

Plain tag names must not contain . or /.

Commands

setup is create-or-update and never destroys an existing tag by default; destructive recreation is explicit (lhd tag reset @tst1 --yes):

lhd setup @tst1 --top foo --files foo.v bar.v
lhd setup @tst1 --set upass.verifier=false        # written into the tag's lhd.toml
lhd setup @exp  --top foo --files foo.v --input baseline=@tst1

run executes one kernel command against the tag:

lhd run compile @tst1 --recipe O2 --emit verilog:net.v
lhd run sim @tst1
lhd run lec @exp --ref @tst1          # impl = the tag's own state; @ref resolves a tag
lhd run pass abc @tst1 --emit-dir lg:net

Each run desugars to exactly one kernel invocation: resolve the tag's files/top/lhd.toml/state dirs into kernel flags, run with --workdir runs/<invocation_id>/logs/ and --result-json, persist --emit-dir state back into the tag, append the envelope to lhd_results.jsonl, advance latest.

status @tag returns one JSON object: config summary, state freshness, last result per command, lock holder. config show|set|validate @tag edits the tag's lhd.toml (the kernel's --config format — lhd describe config).

The kernel's lhd list is global-only (steps, recipes, emit-kinds, error-classes, options, log-channels). The tag layer adds tag-scoped patterns over the tag's persisted lg: state — modules, hierarchy [--top M], stats, history:

lhd list modules @tst1
lhd list hierarchy @tst1 --top foo

Tag directory

tst1/
  lhd.toml                    # tag config: kernel --config format + tag-only keys
  state/lg/                   # persisted graph library between runs
  state/ln/                   # persisted LNAST forest (pre-elaborated imports)
  out/                        # requested --emit outputs (verilog, reports)
  lock                        # per-tag lock for mutating commands
  latest -> runs/20260708T100000Z_7f3a
  runs/
    20260708T100000Z_7f3a/
      lhd_results.jsonl       # appended kernel result envelopes (one per step)
      logs/                   # per-step kernel --workdir logs (NNN_<command>.log)

Locking

Mutating commands (setup, config set, tag reset, run) take the per-tag lock; read-only commands (status, list, config show) do not. A lock failure returns a structured lock_timeout error naming the owning invocation_id; a tag whose state contradicts the request (e.g. lec against a tag whose last compile failed) returns stale_state. Both are tag-layer additions to the kernel taxonomy — lhd list error-classes today ends at unsupported.

Config: lhd.toml

The tag config is the kernel's --config file (strict TOML subset: [pass] tables, key = value, top-level recipe), stored in the tag dir and passed implicitly on every run, extended with tag-only keys:

top = "foo"
files = ["foo.v", "bar.v"]
recipe = "O2"

[inputs]
baseline = "@orig"           # named read-only reference to another tag

[upass]
verifier = false             # any `lhd list options` flag

Precedence: CLI --set (ephemeral, not written back) > tag lhd.toml > built-in defaults. setup is a snapshot: the tag's lhd.toml is self-contained, so later project-default changes do not leak in.

Run log: lhd_results.jsonl

The kernel already emits one JSON result envelope per invocation (--result-json: command, status, run_id, exit_code, inputs, outputs, recipe_steps, error_class/message/hint, n_errors/n_warnings). The tag layer appends that envelope — plus tag, invocation_id, seq, started_at/ended_at/elapsed_ms, log — one line per step to the current run's lhd_results.jsonl. Append-only JSONL: crash-safe, agents read individual lines. lhd run streams the same lines to stdout as steps complete (the kernel's protocol-clean --diag-fmt jsonl rules apply).

The kernel's run_id stays the deterministic content hash (identical rerun ⇒ identical run_id — it is the cache key). Run directories need a name that never collides across concurrent agents, so the tag layer mints the timestamp+random invocation_id and names the run dir after it.

{"schema_version":1,"tag":"@tst1","invocation_id":"20260708T100000Z_7f3a","seq":1,"command":"compile verilog","status":"pass","run_id":"9c81f2…","exit_code":0,"elapsed_ms":1200,"inputs":["foo.v"],"outputs":["state/lg/"],"log":"runs/20260708T100000Z_7f3a/logs/001_compile.log"}
{"schema_version":1,"tag":"@tst1","invocation_id":"20260708T100000Z_7f3a","seq":2,"command":"lec","status":"fail","run_id":"77aa03…","exit_code":1,"elapsed_ms":300,"error_class":"equiv_fail","error_message":"prove_equal: foo mismatch","log":"runs/20260708T100000Z_7f3a/logs/002_lec.log"}

Steps that may hot-reload (run sim @tag --hot) additionally report their tier in the envelope (see the simulation hot-reload tiers in the LiveHD docs) so agents can tell verification-grade from exploration-only runs.
