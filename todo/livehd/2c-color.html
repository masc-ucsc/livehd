2c-color hierarchical node coloring (revive pass/label)

Read a large design as an lgraph hierarchy, walk it from a top, and assign every node an integer color id chosen by a partitioning algorithm. This revives the deleted pass/label suite (four algorithms), but with one core change: the old pass colored a flat graph, whereas this one attaches colors to the hierarchy using the built-in hierarchical iterators. The output is a complete per-node color plus an aggregate color metadata record persisted on the selected top graph.

Scope is coloring + metadata only. Turning the colors into real modules — "one new lgraph per nonzero color id" built into a fresh graph_library — is the separate future PARTITION task (see Pre TODO). This task just produces and records the coloring that PARTITION (and PRPCOLOR, ABC) will consume. The lhd pass color CLI plumbing is in scope; lhd synth can later become a convenience wrapper over a broader lhd pass synth flow.

Depends on

None. The old source lives in git history (see Revival below); the per-node color attribute slot already survives in graph/attrs.hpp.

Background — the pass that was deleted

  • The old suite lived under pass/label/ and was deleted whole in commit e446552a8 "hhds migration" (parent a8b5022e0). It was a casualty of the LGraph→HHDS migration, not an intentional feature removal.
  • Four algorithms, each an Eprp subcommand (pass.label.{mincut,acyclic,synth,path}), all writing an int color via set_color(node, id) (NO_COLOR = 0 = uncolored):
    • mincut — min-cut partitioning via the external VieCut library: gather non-const/non-IO nodes → METIS-format connectivity → viecut_cut(alg) (vc/cactus/…) → colors from the partition. Opts: iters, seed, alg. Reviving this drags VieCut back through MODULE.bazel.
    • acyclic — partition combinational logic into acyclic regions: gather_roots() (outputs + fan-out>1) → grow partitions backward → optional merge_partitions_* → color = partition id. Opts: cutoff, merge. Self-contained (no external dep).
    • synth — forward-propagated synthesis-boundary ids (pipe vs synth mode). Self-contained.
    • path — color nodes adjacent to flops, propagate along FF paths; special-cases clock/reset wire names to avoid false aliases. Self-contained.
  • The per-node color slot already survives the migration: livehd::attrs::color (graph/attrs.hpp:74, flat_storage, comment: "used by pass/label and pass diagnostics") with wrappers has_color / color_of / set_color (graph/node_util.hpp:136,138,291). It replaced Lgraph_attributes::node_color_map.
  • Caveat: the deleted code is pre-hhds (that is why it died in the migration commit). The git source is the algorithm reference, not a clean checkout — node iteration, the Eprp registration, the BUILD deps and the color write all have to be ported to the current hhds::Graph / Node_class API. set_color/color_of already exist for the flat (compact) color attr; the hier_storage variants are new (subtask B).

The delta — flat color → hierarchy-attached color, given a top

Old: color a single flat graph's node table. New: anchor on a designated top graph (resolved by name from the graph_library) and walk the instance hierarchy with the built-in iterators — no flattening:

  • graph->fast_hier() → hhds::FastHierRange and graph->forward_hier() → hhds::ForwardHierRange (../hhds/hhds/graph.hpp:605–606) do the hierarchical traversal automatically. Use those rather than rolling a manual descent or elaborating the design flat.

Color model (agreed semantics)

  • Per-node output is just an integer color id. There is no per-node "shared" flag — sameness of id is the sharing.
  • NO_COLOR = 0 is invalid in a finished coloring. Every partitionable node reachable from the selected top must receive a nonzero color id. A later partition pass must diagnose any remaining zero as a color-pass bug.
  • Same color id ⇒ the same module. A color id is, conceptually, the module that PARTITION will later generate once and instantiate everywhere that id appears.
  • Discontinuous regions may share an id. Two disjoint regions (e.g. two different instances of the same object) carrying the same id means "these are the same module." That is the normal, desirable case.
  • Invariant the coloring must respect: every region that shares a color id must have the same boundary interface and be structurally identical — trivially true when the regions are literally the same module def instantiated more than once. PARTITION relies on this (one generated lgraph per id).
  • Optional continuous mode (the "continuous optional pass" from the COLOR pre-todo): forbid discontinuous same-id regions, splitting a color into per-region ids, for the cases where downstream extraction needs each color to be a single connected region.

Storage — hierarchical color, compact as an option

  • Default: hier_storage. The per-node color is a hier_storage attribute, keyed per hierarchical instance — so two instances of the same module def can carry different ids. This is what "attach the color to the hierarchy" means, and it is the conversion target for every algorithm.
  • Convert the algorithms to hierarchical color. The old algorithms only wrote compact_flat color (per module def). Porting them is therefore not just an API change — each must be reworked to color per hierarchical instance over the fast_hier/forward_hier walk. Add a hier_storage color attr plus hier set_color/color_of variants.
  • Optional compact/flat mode, per pass. When an algorithm guarantees that all instances of a module always share the same color, the existing flat_storage livehd::attrs::color attr (per class-index) is significantly more efficient — one write per def, not per instance. Expose this as a per-pass option; the hierarchical reader falls back to the flat color when no per-instance color is set.
  • One active coloring per top. A color run starts by clearing the active coloring for that top, then the selected algorithm writes the new complete coloring. The algorithm may implement options such as keep_colored, but the final result for the selected top still must be complete and nonzero.

Color metadata (persistent top-graph attr)

Separate from the per-node ids, the pass records an aggregate descriptor of the coloring. It is a persistent HHDS graph attribute attached to the selected top graph, so it is saved inside the graph_library with the graph body. Each top graph has one active coloring record. The attr value is a serialized text blob, preferably one JSON object (not JSONL unless history is added later). Holds hints — not per-node data:

ColoringInfo {
  schema_version : 1
  top         : "top_module"
  algorithm   : "mincut" | "acyclic" | "synth" | "path"
  params      : { ... }              // cutoff, seed, alg, continuous, ...
  colors      : map<color_id, ColorInfo>
}
ColorInfo {
  name        : string               // hint for the generated module name
  region_cnt  : int                  // how many disjoint regions carry this id
  instance_cnt: int                  // how many hierarchical instances it spans
  // ... room for more hints
}

The per-instance hier_storage color is anchored to the top's hierarchy, and the metadata lives on that same top graph. The durable tree-level home for this descriptor is task 3c (Partition descriptor as a tree-level HHDS attribute).

CLI surface

lhd pass color <alg> --top <mod> --set pass.color.<alg>.<param>=v  lg:dir
lhd pass color clear --top <mod>                                  lg:dir
# <alg> ∈ mincut | acyclic | synth | path ; params discoverable via `lhd list options`
# --top selects the hierarchy root; colors + metadata are written in place into lg:dir
# clear removes the active coloring for that top only, not unrelated tops in the library
# --config <file> supplies a bulk option set for complicated runs
# keep_colored: lhd pass color <alg> --top m --set pass.color.<alg>.keep_colored=1 lg:dir

The single color method takes the algorithm as its one positional arg; everything else is --set pass.color.<alg>.*. The matching lhd pass partition verb (task 2p-partition) consumes these colors. Add the lhd pass command, graph-library input/output handling, option listing, and --set pass.color... validation as part of this task.

Subtasks — pending

Core task complete (see landed). Only this non-blocking refinement remains:

  • B′ — per-instance hier_storage write. The hier_color attr + set_hier_color/hier_color_of variants and the unified reader (node_color_of, hier-then-flat) all exist, and the metadata + node_color_of read path already prefer it. But the four algorithms currently write only the flat per-def color: that is correct and far cheaper here because every algorithm colors by graph structure, so all instances of a def share an id (exactly the documented all-instances-share case where flat is the right representation).
