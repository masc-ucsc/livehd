2h-set_path --set/--config namespaces match the command path + context-relative abbreviation

Two coupled CLI changes that apply uniformly to every command and pass option (not just one pass):

  1. Namespace = command path. An option's --set namespace is the command path used to invoke it. lhd pass abc … ⇒ --set pass.abc.…; lhd pass color … ⇒ pass.color.…; lhd lec … ⇒ lec.…. The prefix reads straight off the sub-command and stops passes crowding the base --set namespace.
  2. Context-relative abbreviation. A --set key may drop any leading dotted segment that the command words to its left have already established. So all of these are identical:
lhd --set pass.abc.foo=3 pass abc ...     # before any command word ⇒ must be fully qualified
lhd pass --set pass.abc.foo=3 abc ...     # fully qualified still works after 'pass'
lhd pass --set abc.foo=3 abc ...          # 'pass' seen to the left ⇒ may drop 'pass.'
lhd pass abc --set pass.abc.foo=3 ...     # fully qualified
lhd pass abc --set foo=3 ...              # 'pass abc' seen ⇒ may drop 'pass.abc.'

All five canonicalize to pass.abc.foo=3. The same must hold for every other command/pass option.

Canonical namespace = command path (the registered option name)

  • lhd pass abc … → pass.abc.* (e.g. pass.abc.adder, pass.abc.library).
  • lhd pass color … → pass.color.*; lhd pass partition … → pass.partition.*.
  • lhd lec … → lec.*. lec is a top-level command (like elaborate/check — it is in the opts.command vocabulary, lhd_options.cpp:393), not a pass, so its namespace stays the single token lec.
  • liberty is an inou (lives in inou/), so its namespace follows its inou command path (e.g. inou.liberty.*), not pass.liberty. Confirm the exact invocation when wiring (today it is reached as pass liberty gensim; reconcile).
  • Passes that only run inside recipes (upass, cgen, tolg) have no bare command word to anchor them, so they keep their method short-name namespace as today (--set upass.verifier=…, --set cgen.srcmap=…). Decide whether to also accept a longer path for them; either way the current short forms must keep working.

Current behavior / blocker

  • parse_args (lhd_options.cpp:314–320) collects each --set KEY=VAL into a flat opts.sets (vector<pair<key,value>>) with no record of the command words to its left. The single argv pass already knows the command (opts.command) and routes the pass sub-command word into opts.files (lhd_options.cpp:407–412), but that context is not attached to --set entries.
  • merge_sets / check_known_set_passes (lhd_kernel.cpp:342–400) split a key on the first dot (pass.flag) and reject any second dot as "repeated-pass index addressing." So pass.abc.foo errors today and only the single-token abc.foo works. kSetPasses (~L314) maps short names ({"abc","pass.abc"}, …).
  • --config toml entries are flattened to table.key (lhd_options.cpp:240) and prepended as defaults — they go through the same merge_sets path.

Design

  1. Canonical names. Update kSetPasses + the namespace doc so the registered option namespace is the command path: pass.abc, pass.color, pass.partition, lec, the inou path for liberty, plus existing upass/cgen/tolg. Pass tokens may now contain dots.
  2. Parse-time context capture. In the single argv pass, maintain an ordered list of command-path words seen so far — the command word plus the pass sub-command word (not input positionals like lg:DIR/source files, which are routed to ins/files as data). Stamp each --set/--config entry with a snapshot of that context (extend opts.sets to carry it). --set appearing before any command word gets empty context.
  3. Resolution. To canonicalize a key K with captured left-context words C (dotted), accept the first that resolves against the live registry among:
    • K as-is (already a fully-qualified pass.…flag);
    • join(C[0..k]) + "." + K for k = |C| … 1 (longest prefix first).
    The winner is the fully-qualified name handed to the existing strict typo-check + label merge. No candidate resolves ⇒ the existing unknown-pass/flag error (mention the tried context in the hint). Fully-qualified keys work anywhere (incl. before the command, where C is empty); abbreviations only work once the left-context supplies the dropped prefix.
  4. Last-dot split downstream. merge_sets/check_known_set_passes now operate on canonical fully-qualified names: split on the last dot (flag = final component, pass = everything before). A genuine repeated-index key (upass.2.flag) still fails as unknown pass upass.2 — keep an explanatory message.
  5. --config parity. Toml table entries canonicalize to the same command-path names and run through the same validation. They are context-free (no command words), so a toml table name must be a full command path (e.g. [pass.abc] with adder = "cla"). Document the table naming.

Subtasks — build order

  • 0 — canonical namespaces. kSetPasses + command-path mapping (pass.abc/pass.color/pass.partition; lec top-level; liberty inou path; keep upass/cgen/tolg). Build green.
  • 1 — parse-time context capture. opts.sets carries the left-context command-path words for each entry; the argv loop tracks command + pass sub-command words and excludes input positionals.
  • 2 — resolution + last-dot split. Canonicalize keys via the context-relative rule; switch merge_sets/check_known_set_passes to last-dot split on canonical names.
  • 3 — migrate + surface. Update every existing --set/--config/toml use across tests, fixtures, docs, READMEs, recipes; update lhd list options, help text, and the diag hints that print abc.library=… etc. to the canonical command-path names.
  • 4 — tests. The five equivalent forms above all resolve to the same pass.abc.foo; an abbreviation whose dropped prefix the left-context does not supply errors; typos still error; a fully-qualified key before the command works; --config parity; a representative non-pass command (lec) and an in-recipe pass (upass) both still work.
