2m-multiport_mem multiport / multi-clock memory codegen

The --reader slang front-end lowers the firtool-generated SRAM models cleanly through reader+upass, but multi-clock memories do not yet reach correct Verilog: the gap is in the tolg/cgen memory codegen, not the reader. The driving case is XiangShan's data_mem_8x4 (a 2-read / 1-write memory whose three ports each have their own clock). Make multiport + multi-clock memories emit Verilog and lgcheck-pass against --reader yosys-slang.

Depends on

None.

The failing case

data_mem_8x4 (../xs/build/build_opt/rtl/data_mem_8x4.sv): two read ports clocked by R0_clk / R1_clk and one write port clocked by W0_clk — three distinct clocks, with the write a 2-chunk byte-enable (Memory[W0_addr][32'h0 +: 2] <= W0_data[1:0], etc.).

GEN=../xs/build/build_opt/generated-src
lhd compile --reader slang --top data_mem_8x4 --recipe O1 --emit-dir verilog:impl/ \
  --workdir w -- -I $GEN --define-macro SYNTHESIS ../xs/build/build_opt/rtl/data_mem_8x4.sv

Today this compiles clean (reader + upass: 0 errors, no warnings) and emits complete Verilog, but the result is a silent miscompile: the multi-clock memory is collapsed to the write-port clock — the reader conveys the write clock as the memory's clock_pin, and R0_clk/R1_clk are dropped from the emitted memory instance, so both read ports end up clocked by W0_clk. Wrong Verilog, not a truncated-emit crash.

Subtasks — pending

  • A — per-port memory clock binding (tolg + reader). upass/tolg/upass_tolg.cpp:1273–1274 wires the Memory cell's clock_pin from a single module-wide clock_name_; a memory whose read/write ports use different clocks is silently collapsed to the write-port clock (the reader conveys the write clock as clock_pin; the read-port clocks never reach the memory instance) — a wrong-Verilog miscompile with no diagnostic. The Memory cell already models a per-port clock — clock_pin is pid 2, documented "runtime x 1 or n_ports" (graph/cell.cpp:219) — and cgen already derives single- vs multi-clock by comparing the per-port clock driver pins (cgen_verilog.cpp:393–404). So: have the reader convey each read/write port's clock symbol, and have tolg connect the per-port clock_pin sink (base + 2) for each port instead of one shared clock. Single-clock memories (e.g. a plain reg [3:0] mem[…] on one clk) must stay unchanged.

Verification

For each fixed memory module, compile to Verilog with --reader slang (impl) and --reader yosys-slang (ref) and prove them equivalent with lhd lec --set lec.solver=lgyosys --top <mod> --impl verilog:impl.v --ref verilog:ref.v (the inou/yosys/lgcheck sat/BMC path). data_mem_8x4 must stop being IMPL_FAIL in the XS per-module sweep, and the single-clock chunked memory (already LEC-passing, e.g. the /tmp/m2.sv repro) must not regress.

Design notes

  • Memory cell ports (graph/cell.cpp:217–229): addr(0, ×n_ports), bits(1), clock_pin(2, ×1 or n_ports), din(3), enable(4), fwd(5), posclk(6), type(7; 0 async / 1 sync / 2 array), wensize(8 = #write-enable bits), size(9), rdport(10), init(11); port stride kMemPortStride = 12. Per-port pins live at port_n*12 + offset.
  • How the reference does it: --reader yosys-slang compiles data_mem_8x4 fine — yosys infers the multiport/multiclock $mem_v2 directly, so it is the LEC golden. The yosys-side memory construction is in inou/yosys/lgyosys_tolg.cpp.
  • LEC harness caveat: lgcheck reads both sides with yosys read_verilog -sv; a memory with an async read port + a registered read address can be inconclusive in induction (cf. the soomrv memory-LEC notes), so prefer a hand-written plain-Verilog golden when the source uses constructs yosys cannot import.
