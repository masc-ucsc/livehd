2m-multiport_mem multiport / multi-clock memory codegen

The --reader slang front-end now lowers the firtool-generated SRAM models (byte/chunk-enable writes mem[addr][chunk]<=… via the memory wensize path — landed in 43effbf55 + the constprop reg-memory skip). The reader+upass path is clean, but those memories do not yet reach correct Verilog: the gaps are in the tolg/cgen memory codegen, not the reader. The driving case is XiangShan's data_mem_8x4 (a 2-read / 1-write memory whose three ports each have their own clock). Make multiport + multi-clock memories emit Verilog and lgcheck-pass against --reader yosys-slang.

Depends on

None — the 2s prerequisite landed (the direct --reader slang memory support — chunked wensize writes + module-scope loop-var unroll — that lets these SRAM models reach tolg/cgen at all). This task finishes the codegen side.

The failing case

data_mem_8x4 (../xs/build/build_opt/rtl/data_mem_8x4.sv): two read ports clocked by R0_clk / R1_clk and one write port clocked by W0_clk — three distinct clocks, with the write a 2-chunk byte-enable (Memory[W0_addr][32'h0 +: 2] <= W0_data[1:0], etc.).

GEN=../xs/build/build_opt/generated-src
lhd compile --reader slang --top data_mem_8x4 --recipe O1 --emit-dir verilog:impl/ \
  --workdir w -- -I $GEN --define-macro SYNTHESIS ../xs/build/build_opt/rtl/data_mem_8x4.sv

Today this elaborates clean (reader + upass: 0 errors) but emits truncated Verilog (declarations only, no memory instance, no endmodule): the reader binds the memory to a single global clock, finds none for a 3-distinct-clock memory, warns upass.tolg: memory 'Memory' has no clock input to bind, and then cgen aborts because a sync/async memory with no clock pin is fatal (inou/cgen/cgen_verilog.cpp:406–409).

Subtasks — pending

  • A — per-port memory clock binding (tolg + reader). upass/tolg/upass_tolg.cpp:1273–1274 wires the Memory cell's clock_pin from a single module-wide clock_name_; a memory whose read/write ports use different clocks gets no clock (the "no clock to bind" warn). The Memory cell already models a per-port clock — clock_pin is pid 2, documented "runtime x 1 or n_ports" (graph/cell.cpp:219) — and cgen already derives single- vs multi-clock by comparing the per-port clock driver pins (cgen_verilog.cpp:393–404). So: have the reader convey each read/write port's clock symbol, and have tolg connect the per-port clock_pin sink (base + 2) for each port instead of one shared clock. Single-clock memories (e.g. a plain reg [3:0] mem[…] on one clk) must stay unchanged.
  • B — multiclock cgen_memory wrapper family. Only ware/rtl/cgen_memory_multiclock_1rd_1wr.v exists, and the have_wrapper gate (inou/cgen/cgen_verilog.cpp:422–425) rejects any multiclock memory beyond 1rd_1wr with a mem-wrapper-missing diagnostic. data_mem_8x4 is 2rd_1wr multiclock, so even with subtask A it has no wrapper. Either add the missing cgen_memory_multiclock_<R>rd_<W>wr.v variants (mirroring the single-clock family, which already spans up to 4rd×2wr) or make one parametric multiclock wrapper (per-port rd_clock_N/wr_clock_N, the multiclock template already emits these port names). The wrappers carry WENSIZE/FWD/LATENCY exactly like the single-clock ones so the chunked write still works.
  • C — array_ext.v cgen hang (related, single-clock). ../xs/build/build_opt/rtl/array_ext.v (a 32×74-bit 1-read/1-write SRAM on a single RW0_clk, 37-bit chunked write) elaborates clean through reader+upass (0 errors) but hangs in tolg/cgen on full Verilog emit (no completion at 150 s). A near-identical hand-written minimal (single clock, same 74b/37b chunked write + registered-read-addr async read) compiles to Verilog fine, so the hang is a narrow array_ext-specific cgen path — root-cause it (suspects: cgen on the wide chunked memory, or residual $random/RW0_random logic that survives the SYNTHESIS guard). Listed here because it is the other memory-codegen blocker the same XS SRAM set hits; whoever touches the memory cgen path should fix it together.

Verification

For each fixed memory module, compile to Verilog with --reader slang (impl) and --reader yosys-slang (ref) and prove them equivalent with lhd check --top <mod> --impl verilog:impl.v --ref verilog:ref.v (the inou/yosys/lgcheck sat/BMC path). data_mem_8x4 and array_ext must stop being IMPL_FAIL in the XS per-module sweep, and the single-clock chunked memory (already LEC-passing, e.g. the /tmp/m2.sv repro) must not regress.

Design notes

  • Memory cell ports (graph/cell.cpp:217–229): addr(0, ×n_ports), bits(1), clock_pin(2, ×1 or n_ports), din(3), enable(4), fwd(5), posclk(6), type(7; 0 async / 1 sync / 2 array), wensize(8 = #write-enable bits), size(9), rdport(10), init(11); port stride kMemPortStride = 12. Per-port pins live at port_n*12 + offset.
  • How the reference does it: --reader yosys-slang compiles data_mem_8x4 fine — yosys infers the multiport/multiclock $mem_v2 directly, so it is the LEC golden. The yosys-side memory construction is in inou/yosys/lgyosys_tolg.cpp.
  • LEC harness caveat: lgcheck reads both sides with yosys read_verilog -sv; a memory with an async read port + a registered read address can be inconclusive in induction (cf. the soomrv memory-LEC notes), so prefer a hand-written plain-Verilog golden when the source uses constructs yosys cannot import.
