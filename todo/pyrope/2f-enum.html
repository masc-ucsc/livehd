Group A — enum identity as a match selector / across a boundary

An enum value loses its enum identity when it is a match selector coming from a function parameter or call result, so == Enum.Entry arms fall through to else (the underlying int value is correct; the identity tag is lost).

enum Op = (add, sub, mul)
comb classify(o:Op) -> (r) { r = match o { == Op.add {1} == Op.sub {2} else {0} } }
cassert(classify(o=Op.sub) == 2)   // today: folds to 0 (else)

  • inou/prp/tests/comptime/adv_enum_cast_12.prp

Group B — :int-typed enum sequential numbering

An :int-typed enum (and one continuing after a negative explicit value) must number sequentially (0,1,2,…), not one-hot. The explicit-value form (b=5) already works — only the :int trigger is broken.

enum V:int = (a, b, c)
cassert(int(V.a)==0 and int(V.b)==1 and int(V.c)==2)   // today: one-hot
  • inou/prp/tests/comptime/adv_enum_seq_03.prp
  • inou/prp/tests/comptime/adv_enum_seq_11.prp (negative explicit value continues the sequence)
  • inou/prp/tests/comptime/adv_enum_hier_10.prp (:int suppresses sequential — same root cause)

Group C — enum bitwise union + in membership landed

a | b unions enum members and x in union is bit-containment. This landed with the 2f-enum_hier fix (the membership machinery rides the same enum bit/scalar handling): adv_enum_hier_04.prp and adv_mix_enum_match_08.prp now pass and stay as regression guards.
