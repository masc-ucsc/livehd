2f-unary_type unary operator result loses its bool/int kind tag

A unary operator's result kind is dropped in the typecheck for several shapes, so a parent operator / comparison then rejects a perfectly in-group operand with "requires boolean/integer operands" or "== requires both operands to be the same type". The values would be correct — it never reaches the fold; the kind tag is simply not propagated.

Operator groups (the contract)

  • ~ is integer (bitwise) only; ! / not are boolean only. Each works only within its group.
  • Cross-group use is a compile error and already works correctly today — keep it: not n1, !n1, n1 and true, ~n1 and true (with n1:u3) all error. (n1 != 0) and true, n1 & 4, ~n1 & 3 are the valid spellings.

Depends on

None.

  • Fix only the tests listed below. Do not fix unrelated failing adv_* tests.
  • The fix must not increase the total number of failing tests; in particular it must not loosen the cross-group errors (the lock-in tests below).
  • Comptime / error tests only — no equiv test.

The bug — three shapes (all in-group, all should compile)

// A. boolean unary `not`/`!` loses bool kind  →  rejected by ==/and/or
cassert((not true) == false)        // err: == requires same type
cassert(x or (!y))                  // err: operator or requires boolean operands

// B. nested unary minus loses int kind  →  rejected by `-`
cassert(-(-(-5)) == -5)             // err: operator - requires integer operands
//      (single `-(-5)` works; depth >= 2 breaks)

// C. `~X` loses int kind when feeding a bitwise binary op  →  rejected by &/|/^
const n1:u3 = 1
cassert((~n1 & 3) == ((~n1) & 3))   // err: operator & requires integer operands
cassert(~(a & b) == ((~a) | (~b)))  // De Morgan, untyped consts — same defect

Subtasks — pending

Likely one root cause: the typecheck pass does not stamp the result kind of a unary-op node (or drops it through a grouping/nesting), so the parent op sees a kind-less operand. Fixing kind propagation on the unary node should cover A/B/C.
