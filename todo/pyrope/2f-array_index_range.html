2f-array_index_range custom-range array dimensions [lo..<hi]T

Custom array index domains (mut a:[-4..<4]i5, x4:[100..<132]u3 per 08-memories.md) are unimplemented: the range dimension yields empty dims (no prefill) and the index base offset is never applied, so a[-4] does not map to slot 0.

Scope (applies to every 2f-* adversarial-finding task)

  • Fix only the tests listed below. Do not fix unrelated failing adv_* tests.
  • The fix must not increase the total number of failing tests (run the full inou/prp suite).
  • Comptime / error tests only.

Failing tests

  • inou/prp/tests/comptime/adv_array_basic_06.prp — mut a:[-4..<4]i5 signed-range index domain

Root cause (verified)

extract_array_dims (prp2lnast.cpp:1285) calls resolve_cycle_value which only handles a bare constant/identifier — a range expression_item returns nullopt, so dims is empty (no prefill, and the :[]=nil empty-array path fires). Separately, no index_base offset is subtracted at write (~1972) or read (member_selection_to_node), so a[-4]/a[0]/a[3] are passed verbatim and constprop reports out-of-bounds on the size-2 tuple that the writes grew.

Proposed fix

Make extract_array_dims range-aware: return {size, base} per dim (for [lo..<hi] size=hi-lo, base=lo; plain [N] base=0). Use the size for prefill (stops the empty-array seeding automatically). Record per-variable dimension bases and, at every index site, lower the index as (idx - base) before building the tuple key, with a comptime out-of-bounds check. The enum-index variant ([X]u3) is a separate, larger sub-feature.

Difficulty / risk

Hard (a real feature build, not a one-liner). Medium-to-high risk — touches extract_array_dims, prefill, and every index site; plain [N] arrays must be unchanged (base=0). (Spec: 08-memories.md:349-376.)

Done when

adv_array_basic_06 passes; plain-array tests stay green.
