2f-neg_bitrange negative selector index = compile error (never a segfault); large arithmetic right shift saturates

Two related defects in the low-level shift path (../hlop blop.hpp shln):

  1. A negative bit / array / cycle index — foo#[…], foo[…], foo@[…] whose index value is negative (bw_min < 0), including a decreasing range used as a selector — must be a clean compile error. Today it seg-faults (Assertion failed: (src2 >= 0), function shln, blop.hpp:150).
  2. A large arithmetic right shift must saturate: -1 >> 10000 == -1, 5 >> 10000 == 0, for any shift amount. Today a negative value shifted by a large amount (≳ 64) crashes / errors.

Design decision (range/index semantics)

  • Ranges are always increasing comptime constructs (like tuples). A decreasing range — e.g. 1..=-2, 5..=2 — is a compile error. (Already enforced for a standalone range: "invalid descending range"; the bug is that the bit-select path does not apply this and instead feeds a negative shift amount into shln.)
  • -10..=0 is a valid (increasing) range. But any negative index value taken from it and used in []/#[]/@[] is a compile error.
  • There is no negative / distance-from-end indexing for #[]. Use an open range (b#[1..]) to select "to the end".
