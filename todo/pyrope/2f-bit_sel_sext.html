2f-bit_sel_sext single-bit #[N] sign-extends a set bit to -1

A no-operator (default zext) single-bit selection of a set bit folds to -1 (i.e. 0sb1, sign-extended) instead of the unsigned 1. Zero bits read correctly, so the defect only shows on set bits. The same wrong sign-extension hits #[n..=n], #zext[n], and single-bit selects of typed uN values / literals. The default/zext bit-select path is applying sign extension where the spec mandates zero extension.

Equiv test spec (subtask B)

1-bit zext (a#[3], the buggy path), 2-bit zext (a#[0..=1]), 2-bit sext (a#sext[6..=7], the negative path), combined arithmetically. The result is always non-negative (0..1033) so no signed output port is involved.

// inou/prp/tests/equiv/bit_sel_sext.prp
/*
:name: bit_sel_sext
:type: equiv
:verilog_top: bit_sel_sext.bsel
:pyrope_top: bit_sel_sext.bsel
*/
comb bsel(a:u8) -> (out:u32) {
  mut z1 = a#[3]            // 1-bit zext -> 0 or 1   (NEVER -1)
  mut z2 = a#[0..=1]        // 2-bit zext -> 0..3
  mut s2 = a#sext[6..=7]    // 2-bit sext -> -2..1 (signed)
  out = (z1 * 1000) + (z2 * 10) + (s2 + 2)
}
// inou/prp/tests/equiv/bit_sel_sext.v  (golden — mind Verilog signedness; verify with lgcheck)
module \bit_sel_sext.bsel (
  input  [7:0]  a,
  output [31:0] out
);
  wire        z1 = a[3];                  // 1-bit zext
  wire [1:0]  z2 = a[1:0];                // 2-bit zext
  wire signed [2:0] s2 = $signed(a[7:6]); // sign-extend the 2-bit field -> -2..1
  wire [1:0]  s2p = s2 + 2;               // 0..3, non-negative (no signed output)
  assign out = (z1 * 32'd1000) + (z2 * 32'd10) + s2p;
endmodule
