2f-does_positional positional does matching + named-field declaration kinds

does/equals on tuples with unnamed fields does not enforce positional matching, a value-only RHS field is not kind-checked, and declared int ranges are not carried into overload dispatch. (Two of these tests also use the now-required const/mut field-kind syntax in a tuple-literal declaration.)

Scope (applies to every 2f-* adversarial-finding task)

  • Fix only the tests listed below. Do not fix unrelated failing adv_* tests.
  • The fix must not increase the total number of failing tests (run the full inou/prp suite).
  • Comptime / error tests only.

Failing tests

  • inou/prp/tests/comptime/adv_does_op_08.prp — positional does: (1,b=2) does (b=2,1) wrongly true (BUG A); mutconst syntax migrated
  • inou/prp/tests/comptime/adv_equals_is_has_in_09.prp — LANDED 2026-06-14 (BUG B per-field kind + mutconst syntax)
  • inou/prp/tests/comptime/adv_overload_13.prp — declared int range not used in dispatch: p(100) binds the wrong overload (BUG C)

Landed (2026-06-14)

BUG B + mutconst syntax — structural_does (constprop) now derives each named leaf's kind (from Entry.kind, falling back to the collapsed scalar's repr) and rejects a same-named field of an incompatible kind, so (b:string) does (b=33:int) is no longer a superset. adv_does_op_08 and adv_equals_is_has_in_09 had their (a=1,b=2) tuple-literal declarations migrated to (const a=1, const b=2) for the landed 2f-mutconst rule (matching the 49 tests that commit already patched). adv_equals_is_has_in_09 now passes.

Root cause (verified)

B (easiest): compute_does (upass_constprop.cpp:1984) resolves the SUBSET operand with declared_only=true, so a value-only field b=33 has no kind and string does int isn't rejected. A: structural_does (~966) does not enforce positional matching when an unnamed field is present, because Top_level_entry gives pos=-1 for named fields (the bundle canonicalizes named fields alphabetically, dropping insertion order). C: signature_matches (upass_runner.cpp:3436) uses a power-of-two bits window instead of the declared int(min,max) range, so p(100) mis-dispatches. NOTE: adv_does_op_08/adv_equals_is_has_in_09 now also fail the landed 2f-mutconst rule — their (a=1,b=2) tuple-literal declarations need (const a=1, const b=2). That is test-syntax maintenance for the new rule, separate from the does code bugs (the bare-named-field comparison RHS lines are fine).

Proposed fix

(B) resolve the subset side with declared_only=false (value-inferred kind); apply the same to the sibling case fold at :2402. (A) give Top_level_entry an absolute insertion order and compare named fields by original position once any unnamed field is present (interim: defer rather than return a wrong true). (C) add range_min/range_max to Lnast_io_entry, stash them in upass_ssa type_info_from, and use an exact range test in signature_matches (bits window as fallback). Resolve the 2f-mutconst test-syntax question with the maintainer before editing the two test files.

Difficulty / risk

Medium. B is one line; A touches the widely-used bundle layer; C is additive on Lnast_io_entry. (Spec: 03-bundle.md:179-183, 07-typesystem.md:167-186.)

Done when

All three tests pass (after the field-kind syntax decision); the 1k self/does + 2l enum suites stay green.
