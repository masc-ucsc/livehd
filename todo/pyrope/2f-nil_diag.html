2f-nil_diag illegal ops return nil, not abort; a nil in a constprop output is a diagnostic

✅ LANDED 2026-06-14. Per the user's rule: any arithmetic op that produces a nil is a compile error, and any conditional that uses a nil is a compile error — no exceptions. uPass_constprop::report_arith_nil reports a nil result from the arithmetic ops (+ - * / %, via a report_nil flag on the fold templates — comparisons / logical / casts / bitwise nil-bit propagation legitimately yield nil and are NOT flagged); uPass_runner::report_cond_nil reports a nil if/while condition (match arms keep nil as the legitimate "comptime can't decide" marker). Template bodies are exempt (unbound-param nil placeholders). New error tests adv_nil_operation, adv_nil_cond_if, adv_nil_cond_while; the two adv_div_by_zero* tests still pass; 473/473, no regressions. The addn dest_sz>=1 abort no longer reproduces (the shared-io case folds cleanly now).

Two halves of one mechanism, for reporting bad/illegal operations cleanly instead of crashing or silently folding garbage:

  1. Low-level ops return nil on illegal / unsupported / degenerate input, rather than asserting (Dlop already does this for an illegal 1 << "str"). Today several ops abort the compiler instead.
  2. A nil in an operation / constprop output is reported with a diagnostic ("illegal/unsupported operation"). Today a bad op can fold to a silent degenerate value with no error.

Depends on

None. (hlop/Dlop side of half 1 may be patched in ../hlop.)

Motivating cases

const x = 1 / 0       // today: silently folds to a degenerate value (==0 AND ==1), no error
comb nv(x) -> (x) { x = x + 1 }
comb tw(x) -> (x) { x = x + x }
const r = tw(nv(3))   // today: ABORT  (Assertion (dest_sz >= 1), addn, blop.hpp:48)

Both should instead surface a clean diagnostic: the op yields nil, and a nil reaching a constprop output is reported.
