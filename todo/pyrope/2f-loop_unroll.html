2f-loop_unroll comptime loop unroll defects (one upass-runner mechanism)

All comptime loop unrolling (for, while, loop, break/continue) is handled by the same mechanism in the upass runner (see task_2u migration). Four defects, one engine:

  1. Break-only-exit loops run the body once. while true { … break } and loop { … break } execute the body a single time instead of iterating until the break fires.
  2. continue in a comptime-unrolled for is a no-op — the "skipped" iteration still runs. (break in a for does work, so this is specifically continue.)
  3. for … in X.enumerate() does not unroll — the body runs 0× (plain for v in t works; the .enumerate() desugaring does not).
  4. The while non-termination detector false-positives when the exit is an indirect bool flag rather than a condition variable: a terminating loop is rejected with "comptime loop cannot terminate".

Also seen: combining several of these loop forms in one program aborts the runner (SIGABRT) — fixing the engine should remove the crash; add a regression once it is reproducible standalone.

Examples (all valid; all wrong today)

// 1. break-only exit must iterate until break
mut a = 0
while true { a += 1; if a >= 5 { break } }
cassert(a == 5)                 // today: a == 1

// 2. continue must skip just that iteration
mut s = 0
for i in 0..<5 { if i == 2 { continue }; s = s + i }
cassert(s == 8)                 // today: s == 10 (continue ignored)

// 3. enumerate must unroll per element
mut t = (10,20,30,40); mut sum = 0
for (ix, v) in t.enumerate() { sum = sum + v }
cassert(sum == 100)             // today: sum == 0

// 4. terminating loop with an indirect flag must not be rejected
mut done = false; mut n = 0
while not done { n += 1; if n == 8 { done = true } }
cassert(n == 8)                 // today: "comptime loop cannot terminate"
