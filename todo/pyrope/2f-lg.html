2f-lg lg attribute (explicit lgraph/module name) landed

Landed. Parsing/validation in inou/prp/prp2lnast.cpp (reads ::[lg="…"] from the func_def pipe_config slot, pub-only + comptime-string checks; misplaced lg rejected in reject_common_mistakes_attr_name); the name rides the file-level pub entry (lnast/lnast.hpp Lnast_pub_entry::lg), is copied onto the extracted unit's lg_name_ by upass/func_extract, and becomes the GraphIO/graph name via Lnast::get_graph_name() at every find_io/create_io site in upass/tolg (so cgen emits it and import("lg:…") resolves) while top_module_name — the import("file.entity") key — is untouched. Cross-unit name collisions are diagnosed by uPass_tolg::detect_lg_collisions(). Tests: inou/prp/tests/equiv/lg_rename.prp and inou/prp/tests/errors/lg_not_{pub,string,lambda}.prp.

pub comb foo::[lg="name"] pins the generated lgraph (and hence the netlist/Verilog module name), replacing the compiler-mangled file.entity name. Rules: pub-only, comptime string, lambda kinds only, non-sticky, renames the artifact but never the import key. No code handles the attribute today. Landing this also simplifies 2f-lgimport testing: its Pyrope-produced units can be imported via a pinned lg: name instead of the mangled one.
