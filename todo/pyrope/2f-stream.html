2f-stream streaming prpparse (drop the whole-Ast build)

The prpparse front-end swap (tree-sitter → ../tree-sitter-pyrope/prpparse) landed, but it kept the simple driver: prpparse::Parser::parse_ast() builds the entire file into one global Ast tree, and inou/prp/prp2lnast.cpp walks that whole tree through the tree-sitter-shaped facade (prp_ast_facade.hpp). This task is the remaining performance / memory half: replace the whole-Ast build with a statement stream so the parse tree only ever holds one top-level construct at a time.

The realization (from the original prpparse plan): prp2lnast consumes the parse tree in exactly one place — walk_statement_block(root) inside process_description() — as a single forward walk. Everything else operates on the LNAST. So the whole Ast never needs to exist; the parser can yield one construct, the consumer lowers it to LNAST, and the construct's arena storage is reset before the next one.

Scope (streaming only)

  • Resettable arena. Make Ast_arena reusable: per-construct reset/free that recycles node storage (today it is a monotonically growing std::deque<Ast>).
  • Stream seam in the parser. Add parse_next() / for_each_top_level(consumer) that parses one top-level construct, hands it to a consumer, then resets the arena. The recursive-descent algorithm is unchanged — it already loops over top-level statements; the seam is "yield each one".
  • LiveHD driver change. prp2lnast pulls each construct from the stream (its walk_statement_block driver) instead of iterating a stored description root. Per-node walk logic and all LNAST passes stay as-is; only the top-level driver changes. Resident memory = the current construct's Ast + the growing LNAST (needed anyway).
  • Keep the full-tree consumer. The hhds-tree builder stays the default consumer so the CLI, --sexp, prpfmt, and the differential oracle still see a complete tree and behave identically.

Explicitly out of scope

Large-file splitting / parallel partition parsing (the old plan's "Phase S": parse depth-0 split points concurrently). Streaming already bounds resident memory per construct, so partitioning is only a throughput lever — a separate task if it is ever needed (moot for the current corpus).

Done when

  • prp2lnast lowers from the construct stream; no global parse tree (neither Ast nor the hhds tree) is built on the LiveHD path.
  • All inou/prp tests stay green; prpparse_cli --sexp stays byte-identical for the full-tree consumer.
  • A streaming-consumer unit test asserts resident parse-tree memory stays bounded to one construct on a large concatenated input (e.g. the 1M-line yy.prp stress file from the prpparse perf notes).
  • Record the end-to-end perf delta (materialize / total) vs the whole-Ast build — the win the swap deferred.

Pointers

  • ../tree-sitter-pyrope/prpparse/parser.{hpp,cpp} — parse_ast(), Ast_arena (ast.hpp), parse_description().
  • inou/prp/prp2lnast.cpp — process_description() / walk_statement_block() (the sole parse-tree reader), and prp_ast_facade.hpp (the node facade the consumer binds to).
