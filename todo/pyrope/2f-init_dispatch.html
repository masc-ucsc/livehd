2f-init_dispatch inline anonymous-tuple / kind-aware init constructor dispatch

An inline anonymous-tuple init constructor is not invoked on = value construction, and ctor overload selection is purely arity-based (ignores per-arg kind), so the wrong init body is spliced or none at all.

Scope (applies to every 2f-* adversarial-finding task)

  • Fix only the tests listed below. Do not fix unrelated failing adv_* tests.
  • The fix must not increase the total number of failing tests (run the full inou/prp suite).
  • Comptime / error tests only.

Failing tests

  • inou/prp/tests/comptime/adv_init_ctor_06.prp — int 99 wrongly kind-matches a b:bool ctor param
  • inou/prp/tests/comptime/adv_init_ctor_11.prp — single array/tuple param ctor: construction tuple wrongly exploded into N positional args
  • inou/prp/tests/comptime/adv_mix_deep_fold_21.prp — inline anonymous tuple type carries an init, never dispatched

Root cause (verified)

Three coordinated gaps (06+11 share a runner site; 21 also needs a front-end change):

  • 06: select_init_overload (upass/runner/upass_runner.cpp:3492) picks by arity only — it never runs the per-arg kind/range fit that signature_matches already implements, so init(99) binds a b:bool param.
  • 11: try_init_construction (~upass_runner.cpp:3659) explodes the construction value into N positional args even when the only non-self param is a single array/tuple, so the whole-tuple bind never happens.
  • 21: post-parse LNAST shows named ctor types carry attr_set typename but an inline anonymous tuple type (mut x:(...) = 0) does NOT, so try_init_construction never fires (verifier-off error: cannot assign integer value to constructed (it is tuple)).

Proposed fix

(A) Rewrite select_init_overload to delegate to signature_matches(cio, actuals, /*is_ctor_call=*/true) (kind/range fit at upass_runner.cpp:3427-3447). (B) In try_init_construction, when every init candidate has exactly one non-self, non-scalar param, try a single whole-tuple Ctor_arg binding FIRST; fall back to the exploded multi-arg form only if no overload matches (keeps the (a="x",b=0) multi-arg case at 07-typesystem.md:912). (C) In Prp2lnast::emit_type_spec (prp2lnast.cpp ~4618), when the inline expression_type tuple carries an init field, hoist it to a synthetic anonymous named type and emit the same declare + attr_set typename shape the named path uses, so the runner fires init on the next store. (Option (ii) in the diagnosis — reuses the proven named-type path.)

Difficulty / risk

Medium. The output epilogue / ctor dispatch is on the hot inline path; signature_matches is conservative (no-overload structural fallback rather than wrong dispatch). Verify the full init/ctor/enum + adv_mix_* suites. (Spec: 06-functions.md:494-501, 07-typesystem.md:905-920.)

Done when

All three adv_init_*/adv_mix_deep_fold_21 comptime tests pass; enum payload-init and existing single-overload ctor tests stay green.
