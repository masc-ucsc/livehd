2f-bignum multi-word (>64-bit) integer ops abort / mis-fold

Comptime integer ops on values wider than 64 bits crash or fold wrong. The core arithmetic lives in hlop (blop.hpp / Dlop), which is being patched separately. The LiveHD-side action is to bump the hlop pin to top of tree once the fixes land.

Depends on

hlop patches (in progress, upstream ../hlop).

Scope (applies to every 2f-* adversarial-finding task)

  • Fix only the tests listed below. Do not fix unrelated failing adv_* tests.
  • The fix must not increase the total number of failing tests.
  • Comptime tests only.

Defects

  • Multi-word division aborts — (1<<100)/2, 1e24/1e12 hit Assertion failed: (false && "Multi-word division beyond 64-bit not yet implemented"), divn, blop.hpp:504. (hlop)
  • Signed multiply >64-bit with a negative operand folds wrong — (-big)*(-big) != big*big for big = 1<<90. (hlop muln)
  • Pop-count over a wide all-ones value aborts — #+[0..=127] over (1<<128)-1 hits upass_runner.cpp:942. Root cause: LiveHD is computing the pop-count (and its bitwidth range) itself in lnast/upass instead of delegating to hlop — see subtask B.

Principle: no const ops in LiveHD

LiveHD must not perform pop-count — or any constant-value operation — in lnast/upass. Every Const operation is delegated exclusively to hlop's Dlop (cf. constprop_delegate_to_dlop). The upass_runner.cpp:942 abort exists because LiveHD evaluates/range-checks the pop-count locally; the fix is to route it through Dlop, not to patch a LiveHD-side range.

Tests to fix (already in the regression, currently failing)

  • inou/prp/tests/comptime/adv_int_bigprec_16.prp — >64-bit division (ABORT, hlop).
  • inou/prp/tests/comptime/adv_int_div_10.prp — >64-bit division (ABORT, hlop).
  • inou/prp/tests/comptime/adv_int_bigprec_03.prp — signed >64-bit multiply with a negative operand (wrong fold, hlop).
  • inou/prp/tests/comptime/adv_int_bigprec_10.prp — >128-bit pop-count (ABORT at upass_runner.cpp:942).

Subtasks — pending

  • A — bump hlop to top of tree.
    • Once the hlop multi-word divn / signed muln fixes are pushed, update the git_override commit = for hlop in MODULE.bazel (currently f3f8e44e438d92c01dd3ca877ac165b413984f78) to the latest hlop commit. Rebuild; adv_int_bigprec_16, adv_int_div_10, and adv_int_bigprec_03 should pass.
  • B — delegate the const pop-count to hlop Dlop (don't compute it in LiveHD).
    • adv_int_bigprec_10 aborts in upass/runner/upass_runner.cpp:942 because LiveHD evaluates/range-checks the pop-count locally. LiveHD must do no const-value operation in lnast/upass — route the Const pop-count (and any other const op) through hlop Dlop and drop the local computation/assertion. With wide values handled in hlop, the abort goes away and the fold is correct.

Subtasks — landed

  • A — hlop pin bumped to top of tree (fba3ff2 in MODULE.bazel): fixed multi-word division and signed multiply, so adv_int_bigprec_16 / adv_int_div_10 pass. adv_int_bigprec_03 ((-big)*(-big)) was a separate unary-minus-kind defect, fixed under 2f-unary_type.
  • B — wide pop-count no longer aborts. The value already came from hlop (Dlop::popcount_op); the abort was a LiveHD-side range bug: upass_bitwidth::process_popcount capped the result at the input's get_sbits(), which reports 64 for a >64-bit input (whose int64 Lnast_range collapsed to unbounded) — so a 128-set-bit count violated the [0,64] bw-soundness check at upass_runner.cpp:947. Fix: leave the pop-count range UNBOUNDED for an unbounded input (no false local range; the count rides hlop). All four adv_int_* tests pass, no regressions.
