2f-string_cast string(bool) words; int(true) == -1

Two cast-semantics fixes (rulings from the language owner):

  1. string(true) == "true", string(false) == "false" — render the boolean as its literal word. Today they leak "-1" / "0".
  2. int(true) == -1 (true is the all-ones 1-bit signed value), int(false) == 0. Today int(true) == 1. Consequently string(int(true)) == string(-1) == "-1".

string(<integer>) is decimal text and already works (string(1)=="1", string(-1)=="-1") — keep it. There is no ASCII-byte string(int) decode (see "retired" below).

Depends on

None.

Scope (applies to every 2f-* adversarial-finding task)

  • Fix only the tests listed below. Do not fix unrelated failing adv_* tests.
  • The fix must not increase the total number of failing tests. Blast radius: changing int(true) from 1 to -1 may break in-tree tests/code that assume int(true)==1 — find and update them as part of this task so the net failing count does not grow.
  • Comptime tests only.

  • inou/prp/tests/comptime/adv_int_true_neg.prp — int(true)==-1, int(false)==0, string(int(true))=="-1".
