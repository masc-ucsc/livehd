2f-mutconst a tuple-literal declaration's named field must declare mut/const

When a tuple literal is a declaration — a const/mut binding with no explicit type — every named field must carry its own kind keyword (mut or const). A bare b = 2 is a compile error. This holds for both const and mut bindings (an earlier assumption that const-outer tuples could use bare fields was wrong). The front-end currently accepts the bare form silently.

It is not required where the field kinds come from context — function-call arguments, value comparisons, and assignment to an already-typed variable.

Depends on

None. (Closely related to 2f-splice, which reworks tuple literals; the splice examples use the const/mut field form.)

Correct vs incorrect

// DECLARATIONS (untyped binding) — each named field needs a kind:
const rest = (b = 2, c = 3)                 // ERROR
mut   y    = (ff = 1, zz = 3)                // ERROR
const rest = (const b = 2, mut c = 3)        // OK (the `mut` is allowed even under `const rest`)
mut   y    = (mut ff = 1, const zz = 3)      // OK

// NOT a declaration — kinds come from context, bare fields are fine:
cassert(x == (a = 3, b = 4))                 // OK: value comparison
fcall(x = 3, b = 5)                          // OK: call arguments
mut r:T = (b = 2, c = 3)                     // OK: assignment to a typed variable
const t = (1, 2, 3)                          // OK: positional entries (no names)
